What is a Skill?

A Skill is a self-contained analysis unit that combines a prompt template, an output JSON schema, and the logic to execute it via the Agent Loop. Each Skill focuses on a specific analytical task — like detecting entry points, tracing call graphs, or generating vulnerability hypotheses.

Skills are defined as YAML packages under the skills/ directory:

skills/repo_map/skill.yaml
id: repo_map
name: Repository Map Builder
version: "1.0"
description: >
  Build a comprehensive module map of the codebase,
  detecting the tech stack, directory structure, and key files.

prompt_template: prompt.md.j2
output_schema: schema.json
max_iterations: 15
required_evidence_types:
  - code_snippet
  - config

Skill Details

repo_map

Builds a comprehensive module map of the codebase. Detects the technology stack, directory structure, key configuration files, and main entry points. This is always the first skill to run in any workflow.

ℹ

Output includes: list of modules, detected languages, framework identification, file count by type, and key files worth exploring further.

entrypoints_detection

Finds all HTTP handlers, CLI command definitions, message queue listeners, and background job registrations. These are the external-facing surfaces through which an attacker might interact with the codebase.

Supported frameworks: Express.js, FastAPI, Django, Flask, Spring Boot, Gin, Echo, Rails, NestJS, and more.

call_graph_trace

Traces function call relationships throughout the codebase. Builds a directed graph of which functions call which, enabling downstream analysis of how untrusted input can propagate through the system.

data_structure_extraction

Extracts entity models, database schemas, API request/response types, and configuration structures. Understanding the data model is critical for assessing vulnerability impact.

auth_logic_detection

Locates authentication and authorization guards throughout the codebase. Identifies where user identity is verified, where permission checks occur, and where they might be missing.

input_flow_analysis

Traces how untrusted input flows through the system: from HTTP request parameters → handler → business logic → data access → output. Maps every step with relevant code citations.

sink_detection

Identifies dangerous sink functions — places where untrusted data could cause harm: SQL query execution, OS command running, file I/O, eval(), innerHTML, and similar operations.

CategoryExamples
SQL Injectiondb.execute(), cursor.execute(), Query.raw()
Command Injectionexec(), system(), eval(), child_process.exec()
XSSinnerHTML, dangerouslySetInnerHTML, render()
Path Traversalopen(), readFile(), send_file()

dependency_analysis

Maps third-party dependencies from package.json, requirements.txt, go.mod, Cargo.toml, pom.xml, and similar files. Checks for known vulnerable versions.

attack_surface_mapping

Correlates entry points, sinks, and input flows to produce a risk-ranked attack surface map. Each surface is scored based on reachability, impact, and existing mitigations.

vulnerability_hypothesis

Generates CWE-tagged vulnerability hypotheses based on all prior evidence. Each hypothesis includes the CWE type, affected components, and confidence score.

CWEVulnerability Type
CWE-89SQL Injection
CWE-78OS Command Injection
CWE-79Cross-Site Scripting (XSS)

exploit_generation

Generates proof-of-concept concepts (not functional exploit code) for validated vulnerability hypotheses. Each PoC includes the attack vector, required conditions, and expected impact.

report_generation

Synthesizes all findings from previous steps into a structured Markdown report. Each finding is linked back to MEU evidence citations for full traceability.

SkillAgent

Each Skill is executed by a SkillAgent — a wrapper around the AgentLoop that provides the Skill's prompt template, input schema, and output schema.

agent/skill_agent.py
from agent.skill_agent import SkillAgent

agent = SkillAgent(
    skill=skill_pkg,
    llm_adapter=llm,
    tool_executor=tool_executor,
    evidence_store=evidence_store,
    memory_store=memory_store,
    loop_config=loop_config,
)

result = agent.run(inputs={"repo_path": "/path/to/repo"})
# result.skill_output — the structured JSON output from the skill
# result.evidence_stored — list of MEU IDs extracted
# result.loop_result — LoopResult with iteration stats

Creating a Custom Skill

To create a new Skill, add a package under skills/ with three files:

skills/my_skill/skill.yaml
id: my_skill
name: My Custom Skill
version: "1.0"
description: >
  Brief description of what
  this skill analyzes.

prompt_template: prompt.md.j2
output_schema: schema.json
max_iterations: 10
skills/my_skill/schema.json
{
  "type": "object",
  "properties": {
    "findings": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "claim": {"type": "string"},
          "evidence_refs": {
            "type": "array",
            "items": {"type": "string"}
          },
          "confidence": {"type": "number"}
        },
        "required": ["claim", "evidence_refs"]
      }
    }
  }
}

The Jinja2 prompt template (prompt.md.j2) receives these variables:

VariableDescription
{{ repo_path }}Absolute path to the repository
{{ inputs.* }}Workflow input parameters
{{ steps.*.outputs.* }}Outputs from previous workflow steps
{{ context.session_id }}Current analysis session ID
