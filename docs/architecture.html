System Overview

Odin is built as a layered system where each phase builds on the previous one. The architecture spans from low-level tool execution through to high-level research workflows — all unified by a shared evidence model.

The system processes a code analysis request through this flow:

Layer 0
CLI Entry FastAPI Server od in.py
↓
Layer 1
WorkflowOrchestrator PipelineExecutor
↓
Layer 2
SkillAgent AgentLoop
↓
Layer 3
ToolExecutor LLMAdapter
↓
Layer 4
EvidenceStore MemoryStore RAGStore
↓
Layer 5
Skill Prompts YAML Workflows JSON Schema

Phase 1 — Tool Executor

The Tool layer gives the LLM real "hands" to interact with the codebase. Each tool follows the Tool Protocol interface and returns a ToolResult regardless of success or failure.

Tool Description Safety
read_file Read file content with line range support Safe
list_dir Directory tree listing with file type icons Safe
search_code Regex code search with configurable context lines Safe
run_shell Safe shell commands (whitelist: git, find, grep, wc...) Whitelisted
git_clone Clone remote repository to local temp directory Safe
git_log View commit history Safe
git_diff View file changes between commits Safe
detect_lang Detect programming language and framework stack Safe

Tools are registered using the @tool decorator and automatically exposed to the LLM in OpenAI Function Calling format.

ℹ

Security boundary: run_shell uses a strict command whitelist. The LLM cannot execute arbitrary shell commands. read_file enforces path resolution against the repo root to prevent traversal attacks.

Phase 2 — Agent Loop

The Agent Loop is the core execution engine that enables iterative LLM ↔ tool interaction. It follows the classic ReAct pattern:

Render Skill Prompt → Call LLM → Has tool_calls? → Extract Output ✓

When tools are requested, the loop executes them and feeds results back:

Execute Tools → Append ToolMessage → Continue LLM Loop

The loop terminates when:

  • LLM returns plain text (no tool calls) → Success
  • Max iterations reached → max_iterations status
  • Consecutive tool errors exceed limit → Graceful degradation or failure

Loop Configuration

Config Field Default Description
max_iterations 20 Max LLM → tool iterations per SkillAgent run
evidence_required true Enforce evidence citations in all conclusions
require_final_json true Require the final LLM output to be valid JSON
allow_fallback_on_error true Continue even if individual steps fail

Phase 3 — LLM Adapter

The LLMAdapter abstract base class provides a unified interface for all LLM providers. All adapters return a normalized LLMResponse regardless of the underlying API:

OpenAI Adapter

Supports GPT-4o, GPT-4o-mini, and any OpenAI-compatible API endpoint.

openai.py
llm = create_adapter(
    "openai",
    default_model="gpt-4o-mini"
)

Anthropic Adapter

Supports Claude 3.5 Sonnet, Claude Sonnet 4, and Claude 3 Opus.

anthropic.py
llm = create_adapter(
    "anthropic",
    default_model="claude-sonnet-4"
)

Ollama Adapter

Connects to local Ollama instances with OpenAI-compatible API.

ollama.py
llm = create_adapter(
    "ollama",
    base_url="http://localhost:11434/v1",
    default_model="llama3"
)

Mock Adapter

Simulates LLM responses for testing without API costs.

mock.py
llm = create_adapter("mock")
# Returns preset responses
# or simulated tool calls

Phase 4 — Pipeline Executor (DAG Parallelism)

The Pipeline Executor takes a workflow definition and schedules its steps across DAG layers using Kahn's algorithm. Steps within the same layer run concurrently via ThreadPoolExecutor.

DAG Layering Example (vulnerability_research)

Layer 1
repo_discovery
↓
Layer 2
entrypoints_detection sink_detection dependency_analysis
↓
Layer 3
call_graph_trace auth_logic_detection
↓
Layer 4
input_flow_analysis
↓
Layer 5
attack_surface_mapping
↓
Layer 6
vulnerability_hypothesis
↓
Layer 7
exploit_generation
↓
Layer 8
report_generation
ℹ

In Layer 2, all three steps (entrypoints_detection, sink_detection, dependency_analysis) run in parallel threads. This means a 3-core machine completes Layer 2 in roughly 1/3 the time of sequential execution.

Phase 5 — Memory & Storage

Odin maintains three interconnected stores that preserve analysis state across runs:

EvidenceStore

Stores MEUs (Minimum Evidence Units) with full code citations. Indexed by file path, symbol, and tags for fast retrieval.

MemoryStore

Stores ResearchArtifacts and Conclusions with cross-references. Enables the system to build on previous findings.

RAGStore

SQLite FTS5 full-text search over historical reports. Past analyses inform future ones automatically.

Phase 6 — MCP Integration

The Model Context Protocol (MCP) client allows Odin to connect to external services for enriched analysis:

Integration Capabilities
mcp_github Fetch issues, PR reviews, repository metadata, and commit history
mcp_cve Cross-reference findings against CVE databases and known vulnerability patterns
mcp_jira Correlate security findings with existing tickets and bug reports
custom Extend with your own MCP servers for domain-specific integrations

Data Flow

A complete analysis request flows through the system as follows:

data_flow.txt
1. CLI: python odin.py analyze ./repo --workflow vulnerability_research
          ↓
2. GitOps: Clone repo (if remote) → local temp directory
          ↓
3. WorkflowOrchestrator: Load YAML definition
          ↓
4. PipelineExecutor: Group steps into DAG layers (Kahn's algorithm)
          ↓
5. For each layer (parallel where possible):
     a. SkillAgent: Receive skill prompt + repo context
     b. AgentLoop: Iterative LLM → tool → LLM → ... → output
     c. ToolExecutor: read_file / search_code / git_ops / run_shell
     d. EvidenceStore: Extract & store MEUs with code citations
          ↓
6. RAGStore: Index report for future retrieval
          ↓
7. Output: Structured Markdown report with evidence-linked findings
