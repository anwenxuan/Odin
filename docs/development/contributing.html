Development / Contributing

Contributing

Help build the future of systematic vulnerability research. Every contribution matters.

Odin is an open-source project and we welcome contributions of all kinds — from bug reports and documentation improvements to new skills and core features.

Ways to Contribute

Bug Reports

Found a bug? Open an issue on GitHub with reproduction steps, expected vs actual behavior, and your environment details.

Feature Requests

Have an idea for a new Skill, workflow, or integration? Open a GitHub Discussion to propose it before writing code.

Pull Requests

Fix bugs, implement features, or improve tests. All PRs go through review — see the PR workflow below.

New Skills

Extend Odin's capabilities with custom Skills. See the New Skill Guide for the full implementation framework.

Documentation

Improve docs, fix typos, add examples, or translate guides. Documentation contributions are always highly appreciated.

Pull Request Workflow

Before writing code for a new feature, open a GitHub Discussion to align on the approach. This saves you time and ensures your work aligns with Odin's architecture.
  1. Fork the repository
    Fork anwenxuan/Odin to your own GitHub account.
  2. Clone your fork
    Clone the forked repository and set up your development environment following the Dev Setup guide.
  3. Create a feature branch
    Branch naming convention:
    feat/<short-description> — new features
    fix/<short-description> — bug fixes
    docs/<short-description> — documentation
    skill/<skill-name> — new skills
    refactor/<short-description> — refactoring
  4. Make your changes
    Implement your change. Keep commits focused and atomic — one logical change per commit.
  5. Run tests and linters
    Before pushing, verify everything passes:
    bash 
    pytest tests/ -v
ruff check .
mypy .
  6. Submit a Pull Request
    Open a PR against main on anwenxuan/Odin. Fill in the PR template completely — describe the why, not just the what. Link any related issues.
  7. Review & iterate
    Maintainers will review your PR. Address feedback by pushing additional commits — do not force-push during review unless asked.

Code Standards

Python Style

Odin uses ruff for linting and formatting. Configuration lives in pyproject.toml.

python 
# Good: type hints on all public functions
def analyze_vulnerability(
    repo_path: str,
    workflow: str = "vulnerability_research",
) -> AnalysisResult:
    """Analyze a repository for vulnerabilities using the specified workflow."""
    ...

# Bad: missing type hints on public API
def analyze_vulnerability(repo_path, workflow="vulnerability_research"):
    ...

Commit Message Format

Odin follows the Conventional Commits specification:

bash 
# Format: type(scope): description
feat(skills): add SQL injection detection skill
fix(core): correct MEU evidence validation threshold
docs(contributing): add new skill framework guide
refactor(workflow): extract DAG execution into separate module
test(auth): add integration tests for guard analysis
chore(deps): bump ruff from 0.1.0 to 0.2.0

# Allowed types: feat | fix | docs | style | refactor | test | chore | perf | ci
Breaking changes must include BREAKING CHANGE: in the footer or ! after the type (e.g., feat(core)!:).

Adding a New Skill

A Skill in Odin is a self-contained unit of capability. Each Skill lives in odin/skills/<category>/ and follows a strict interface contract.

Skill Structure

bash 
odin/skills/
├── analysis/           # Analysis-layer skills
│   └── sql_injection/
│       ├── __init__.py
│       ├── detector.py   # Core detection logic
│       ├── rules.py      # Pattern matching rules
│       └── skill.yaml    # Skill metadata
└── collection/          # Collection-layer skills

Skill Metadata (skill.yaml)

yaml 
name: sql_injection
version: "1.0.0"
category: analysis
layer: layer_3_vulnerability_scan
description: "Detect SQL injection vulnerabilities via taint analysis and pattern matching."
meu_enforced: true
dependencies:
  - entrypoints
  - call_graph
parameters:
  sensitivity:
    type: string
    enum: [low, medium, high]
    default: medium
  context_lines:
    type: integer
    default: 3
evidence_schema:
  type: object
  properties:
    vulnerability_type:
      type: string
      enum: [sql_injection, blind_sql,_time_based]
    confidence:
      type: number
      minimum: 0
      maximum: 1
    severity:
      type: string
      enum: [critical, high, medium, low]

Skill Implementation (detector.py)

python 
from typing import Any
from odin.skills.base import BaseSkill, SkillResult
from odin.evidence import MEUUnit


class SQLInjectionDetector(BaseSkill):
    """Detects SQL injection vulnerabilities using taint analysis."""

    skill_name = "sql_injection"
    layer = "layer_3_vulnerability_scan"

    def __init__(self, config: dict[str, Any] | None = None):
        self.sensitivity = config.get("sensitivity", "medium") if config else "medium"
        self.context_lines = config.get("context_lines", 3)

    async def execute(self, context: dict[str, Any]) -> SkillResult:
        results = []
        call_chains = context.get("call_chains", [])
        entry_points = context.get("entry_points", [])

        for chain in call_chains:
            finding = self._analyze_chain(chain, entry_points)
            if finding:
                # Create MEU-compliant evidence unit
                meu = MEUUnit(
                    skill=self.skill_name,
                    evidence=finding["evidence_snippet"],
                    metadata=finding,
                )
                results.append(meu)

        return SkillResult(
            skill=self.skill_name,
            findings=results,
            summary=f"{len(results)} potential SQL injection findings",
        )

Registering the Skill

python 
# odin/skills/__init__.py
from odin.skills.analysis.sql_injection.detector import SQLInjectionDetector

SKILL_REGISTRY: dict[str, type[BaseSkill]] = {
    "sql_injection": SQLInjectionDetector,
    "xss": XSSDetector,
    "auth_guard": AuthGuardAnalyzer,
    # ... other skills ...
}
MEU enforcement — Every skill must produce MEUUnit evidence objects. Findings not backed by traceable evidence will be rejected by the evidence enforcement layer.

Code of Conduct

Odin follows the Contributor Covenant Code of Conduct v2.1. By participating, you agree to uphold a welcoming, respectful, and harassment-free environment for everyone.

Short version: Be kind. Assume good faith. Disagree respectfully. Focus on the work, not the person.

PR Checklist

  • All new code has type hints on function signatures
  • ruff check . passes with no errors
  • mypy . passes with no errors
  • pytest tests/ -v passes with new tests added
  • Commit messages follow Conventional Commits format
  • PR description explains the why, not just the what
  • Breaking changes are explicitly marked
