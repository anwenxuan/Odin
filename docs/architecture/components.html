Core Components

A detailed reference for every module in the Odin system. Each entry covers responsibilities, key interfaces, and how it interacts with the rest of the stack.

Layer 2

Agent Loop

The autonomous reasoning engine. Orchestrates LLM calls, monitors convergence, manages checkpoints, and handles retries — all in a tight iterative loop until the task is complete or a termination condition is met.

Loop Controller

The conductor. Manages max iterations, timeouts, early termination signals, and convergence detection. Evaluates whether the agent has reached a stable answer state after each iteration.

Observer

Watches intermediate agent state — tool call history, MEU candidates, evidence gaps. Emits structured events consumed by the evaluation and logging subsystems. Supports real-time UI feedback.

Checkpoint

Serializes agent state after each completed iteration — Working Memory, tool call log, MEU buffer, DAG progress. Enables crash recovery and human-in-the-loop review at any iteration boundary.

Retry Handler

Implements exponential backoff for transient LLM API failures (rate limits, network timeouts). Falls back to a lower-cost model if the primary model is unavailable. Max retry count and backoff schedule are configurable.

Python — AgentLoop interface 
class AgentLoop:
    def run(task: TaskContext, max_iter: int = 20, timeout: float = 300.0) -> TaskResult:
        """Main entry: runs the agent loop until convergence or termination."""
        ...

    def checkpoint() -> CheckpointSnapshot:
        """Returns a serializable snapshot of current loop state."""
        ...

    def restore(snapshot: CheckpointSnapshot) -> None:
        """Restores loop state from a checkpoint snapshot."""
        ...

Layer 1 — Tools

Tool Executor

The runtime for all skill operations. Manages tool invocation, sandboxing, timeout enforcement, and structured result serialization. Acts as the bridge between the Agent Loop's reasoning decisions and the actual filesystem/network operations.

Responsibilities

  • Dispatch tool calls to the appropriate executor (local shell, MCP, or sandboxed worker)
  • Enforce per-call timeout, memory limit, and output truncation
  • Serialize results into a normalized ToolResult schema with error codes
  • Maintain a call log for the Observer (instrumented, not blocking)
  • Support streaming output for long-running commands

Key Interfaces

class ToolExecutor:
    async def execute(
        tool_call: ToolCall,
        sandbox: bool = False
    ) -> ToolResult:
        ...

class ToolCall(BaseModel):
    name: str
    args: dict[str, Any]
    timeout: float = 30.0
    sandbox: bool = False

class ToolResult(BaseModel):
    stdout: str
    stderr: str
    exit_code: int
    duration_ms: float
    error: str | None

Layer 1 — Tools

Skill System

Skills are Odin's atomic analysis units — self-contained, composable modules that the Agent Loop calls to perform specific tasks. Every skill produces structured output and can be wired into any workflow via the DAG definition.

Python — Skill base interface 
class BaseSkill(ABC):
    name: str
    description: str
    input_schema: type[BaseModel]
    output_schema: type[BaseModel]
    requires: list[str]   # skill dependencies
    provides: list[str]    # artifact keys

    async def execute(ctx: SkillContext, input: BaseModel) -> SkillOutput:
        raise NotImplementedError

class SkillRegistry:
    def register(skill: type[BaseSkill]) -> None
    def resolve(dag: DAGDefinition) -> list[list[SkillNode]]
    def get(name: str) -> type[BaseSkill]

Layer 3 — Core

Workflow Orchestrator

Selects the appropriate skill DAG for a given task, resolves dependencies, and hands off the layered execution plan to the Pipeline Executor. Also handles dynamic workflow switching when the agent decides mid-task that a different strategy is needed.

Responsibilities

  • Match task intent to a named workflow (e.g., vulnerability_research, audit)
  • Topologically sort the skill DAG and compute parallel layers via Kahn's algorithm
  • Validate that all required skills are registered and not in conflict
  • Support workflow composition and conditional branching based on intermediate results

Key Interface

class WorkflowOrchestrator:
    def plan(
        task: TaskContext
    ) -> list[list[SkillNode]]:
        """Returns a list of execution layers."""
        ...

    def execute_plan(
        layers: list[list[SkillNode]],
        ctx: SkillContext
    ) -> WorkflowResult:
        """Runs layers sequentially; parallel within layer."""
        ...

Layer 4 — Memory

Memory System

Odin's memory subsystem is split into four stores, each serving a distinct purpose. They share no internal state and can be queried and updated independently, preventing cross-contamination of concerns.

Working Memory

In-memory, session-scoped store for the current task's context. Holds intermediate agent reasoning, partial findings, tool call buffers, and LLM context windows. Cleared at task end.

interface: MemoryKV

Long-term Memory

Persistent, JSON-serialized fact store keyed by entity (file, function, CVE). Survives sessions. Used to recall conclusions from previous analysis runs on the same codebase.

storage: SQLite (row-based)

Evidence Store

MEU

The heart of MEU enforcement. Every verified finding is stored as a Minimum Evidence Unit: file path, line range, and code snippet. No MEU means no conclusion in the final report.

storage: SQLite (row-based)

RAG Store

Full-text search over historical analysis reports using SQLite FTS5. Enables natural-language querying of past findings: "any SQL injection in the auth module?" Returns ranked snippets with source citations.

storage: SQLite FTS5

Layer 2 — Agent Loop

LLM Adapter

A thin abstraction over LLM providers. Normalizes API differences (OpenAI, Anthropic, Ollama) into a single LLMResponse interface. Handles prompt templating, token counting, streaming, and structured output via response models.

Supported Providers

OpenAI gpt-4o, gpt-4o-mini, gpt-4-turbo, gpt-3.5-turbo
Anthropic claude-3-5-sonnet, claude-3-opus, claude-3-haiku
Ollama llama3, codellama, mistral, deepseek-coder (local)

Key Interface

    def max_context(model: str) -> int

Layer 1 — Tools

MCP Client

Connects Odin to external data sources via the Model Context Protocol. Tools are loaded dynamically from MCP servers — no code changes required to add a new external integration. Built-in MCP clients for GitHub, CVE databases, and Jira.

GitHub Integration

Fetch repo metadata, issues, PR diffs, commit history, and file contents. Enriches context with real external data during analysis.

CVE Database

Cross-reference findings against known vulnerability databases (NVD, OSV). Enriches vuln_fingerprint skill output with severity scores, affected versions, and patch links.

Jira Integration

Create or update Jira tickets from confirmed findings. Links evidence MEUs directly to ticket descriptions, making vulnerability reports immediately actionable.

Custom MCP Servers

Add any MCP-compatible tool server via configuration. Odin automatically discovers tools and exposes them to the Agent Loop as first-class skills.

Infrastructure

Evaluation Framework

Quantifies Odin's accuracy using precision, recall, and F1 against labeled ground truth datasets. Runs offline after a task completes. Results are persisted to SQLite and exportable as OpenTelemetry traces.

Metrics Computed

  • Precision — fraction of reported findings that are true positives
  • Recall — fraction of ground truth findings that were detected
  • F1 Score — harmonic mean of precision and recall
  • MEU Coverage — percentage of conclusions with valid MEU attached
  • Hallucination Rate — conclusions with no MEU / total conclusions

Key Interface

class EvaluationFramework:
    def evaluate(
        predictions: list[EvidenceUnit],
        ground_truth: list[EvidenceUnit],
    ) -> EvalResult:
        """Returns precision, recall, F1, MEU coverage."""
        ...

class EvalResult(BaseModel):
    precision: float
    recall: float
    f1: float
    meu_coverage: float
    hallucination_rate: float
    false_positives: list[EvidenceUnit]
    false_negatives: list[EvidenceUnit]

User Entry Point

FastAPI Server

The HTTP gateway. Exposes a REST API for programmatic access to all Odin capabilities — task submission, status polling, result retrieval, and evidence querying. Supports async long-running tasks via background job IDs.

Key Endpoints

# Task lifecycle
POST   /tasks                  # Submit analysis task
GET    /tasks/{id}             # Poll task status
DELETE /tasks/{id}             # Cancel running task

# Results
GET    /tasks/{id}/report      # Download full report
GET    /tasks/{id}/evidence    # List all MEU evidence

# RAG / Knowledge
GET    /search?q=...           # Query historical reports

# Evaluation
POST   /eval/run               # Trigger benchmark run
GET    /eval/results           # Retrieve eval metrics

# Config
GET    /config/workflows       # List available workflows
GET    /config/skills          # List registered skills

Key Features

  • Async task queue — long-running analyses return a job ID immediately
  • Bearer token authentication with configurable scope
  • OpenAPI 3.1 auto-generated docs at /docs
  • Structured logging (JSON) with request tracing via OpenTelemetry
  • CORS support for cross-origin SDK / web UI integration
