System Architecture

A four-layer autonomous system for evidence-backed code analysis. From raw tools to reasoning agents to structured memory — every component traceable.

Architecture Diagram

Full System Stack

Six layers from user entry to infrastructure, connected by a controlled data flow.

Layer Responsibilities

Each layer, clearly defined

Odin uses a strict layered architecture where each layer only communicates with adjacent layers. This separation ensures testability, extensibility, and predictable performance.

0

User Entry Point

Accepts natural-language tasks from the CLI or REST API. Parses arguments, validates credentials, and forwards structured task definitions to the Agent Loop. Acts purely as a thin gateway — no business logic lives here.

1

Tools Layer

12 built-in skills covering repo mapping, entry point detection, call graph analysis, dependency enumeration, and more. Each tool is self-contained and MCP-compatible. External MCP tools are loaded dynamically without modifying core code.

2

Agent Loop

The autonomous reasoning core. The LLM Adapter normalizes provider differences (OpenAI, Anthropic, Ollama). The Loop Controller manages iteration, termination, and convergence checks. The Observer monitors intermediate state. Checkpoint enables resume on failure. Retry handles transient errors with exponential backoff.

3

Core Execution

The execution engine. Pipeline Executor resolves skill DAGs and dispatches tasks to ThreadPoolExecutor for parallel layer execution. Workflow Orchestrator selects and sequences skills. Skill Loader manages the registry and handles hot-reload. Prompt Runner assembles context windows with token budget management.

4

Memory System

Four distinct memory stores: Working Memory (session-scoped, ephemeral), Long-term Memory (persistent JSON facts), Evidence Store (MEU-compliant findings with file/line/snippet), and RAG Store (SQLite FTS5 full-text search over historical reports). Each store has a dedicated access API with no cross-contamination.

I

Infrastructure

Sandboxing isolates untrusted code execution. Verification engine cross-checks findings against ground truth or static analysis tools. Evaluation framework measures precision, recall, and F1 against labeled datasets. Observability layer exports traces, metrics, and structured logs via OpenTelemetry-compatible interfaces.

Execution Flow

End-to-end task lifecycle

From a single command to a structured evidence-backed report, every step is deliberate and traceable.

  1. User Input
    The user runs odin analyze <target> via CLI or sends a POST request to the REST API. The entry point parses the target, selects a workflow (e.g., vulnerability_research), and creates a TaskContext object with session metadata.
  2. Workflow Initialization
    The Workflow Orchestrator loads the named workflow definition from the skills registry. It resolves the skill DAG — determining which skills can run in parallel (same layer) and which must wait for upstream dependencies. This produces a sorted execution plan.
  3. DAG-Layer Parallel Execution
    The Pipeline Executor iterates over each layer. Within a layer, all ready skills are dispatched concurrently via ThreadPoolExecutor. Results flow into the next layer's input. A 7-step workflow that would take 7 × N seconds sequentially might complete in 3 layers × N seconds in parallel.
  4. Agent Loop Reasoning
    As each skill executes, the Agent Loop monitors intermediate findings. The LLM Adapter assembles a context prompt from Working Memory, the current skill output, and relevant Evidence Store entries. The LLM generates reasoning steps, tool calls, and MEU candidates.
  5. MEU Enforcement
    Every LLM-generated conclusion must be paired with a Minimum Evidence Unit — a file path, line range, and code snippet. The Schema Validator rejects conclusions lacking MEU. Unverified claims are flagged as UNVERIFIED and excluded from the final report.
  6. Evidence Store & RAG Indexing
    Verified MEU entries are persisted to the Evidence Store (SQLite). Concurrently, the report text is chunked and indexed into the RAG Store (SQLite FTS5). Future queries can retrieve relevant historical findings via semantic or keyword search.
  7. Report Generation
    The Prompt Runner assembles the final report from the verified Evidence Store entries, organized by severity and skill. The output is written as Markdown to .odin/audit/ with a structured JSON manifest for programmatic consumption.
  8. Evaluation (Optional)
    If a ground truth dataset is provided, the Evaluation Framework computes precision, recall, and F1 for this run. Results are appended to a metrics SQLite file and exported as an OpenTelemetry trace if observability is configured.
