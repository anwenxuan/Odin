Data Flow

Visual diagrams and step-by-step walkthroughs of Odin's three core pipelines: request analysis, tool execution, and evidence storage.

Data Flow Diagram Analysis Request Tool Calling Evidence Storage

Data Flow Diagram

Master Pipeline Overview

Three interlocking pipelines — request, execution, and evidence — share a common memory layer and converge at the final report.

1

Pipeline 1

Analysis Request Flow

How a single user command becomes a structured, evidence-backed analysis report.

  1. User Issues Command
    The user runs odin analyze https://github.com/owner/repo --workflow vulnerability_research via the CLI, or POSTs to /tasks via the REST API. The request includes the target (local path, git URL, or archive), the named workflow to apply, and optional overrides (provider, model, skill filters).
  2. FastAPI Server Receives and Validates
    The FastAPI server parses the request body, validates credentials (API key or bearer token), checks the target accessibility (clone if git URL, verify path if local), and creates a TaskContext object. The task is assigned a UUID and placed in the async task queue. A 202 Accepted response returns the task ID immediately.
  3. Workflow Orchestrator Plans the DAG
    The Workflow Orchestrator loads the named workflow definition from the skills registry. It resolves the dependency graph — for vulnerability_research, this means: repo_discovery (layer 1), then entrypoints + call_graph + dependency_enum (layer 2), then auth_guard_analysis + data_flow + vuln_fingerprint (layer 3), then report_compiler (layer 4). The DAG is topologically sorted using Kahn's algorithm to produce a layer-ordered execution plan.
  4. Pipeline Executor Runs DAG Layers
    The Pipeline Executor iterates over each layer. Skills within the same layer run concurrently via concurrent.futures.ThreadPoolExecutor. Each skill receives a SkillContext with read access to the outputs of all upstream skills. Layer outputs become the shared inputs for the next layer. A 7-skill workflow that would take 7 × N seconds sequentially may complete in 3 × N seconds with parallel layers.
  5. Agent Loop Monitors and Reasons
    As skills execute, the Agent Loop observes intermediate results via the Observer. The LLM Adapter assembles a context prompt from Working Memory (current findings), the active skill output, and relevant Long-term Memory entries. The LLM generates reasoning steps, MEU candidates, and tool call suggestions. The Loop Controller evaluates convergence after each skill layer completes.
  6. Report Compiler Assembles Output
    Once all skill layers complete, the report_compiler skill reads all verified Evidence Store entries (those with MEU), organizes them by severity (Critical → High → Medium → Low), and formats them into Markdown with embedded file/line citations. A structured JSON manifest is generated alongside it for programmatic consumers.
  7. Report Persisted and Returned
    The Markdown report is written to .odin/audit/report-{date}.md. The JSON manifest is written to .odin/audit/report-{date}.json. Both are stored in the task output directory. The FastAPI server marks the task as completed in the queue, making the report available via GET /tasks/{id}/report.
2

Pipeline 2

Tool Calling Flow

How the Agent Loop decides to call a tool, how the Tool Executor runs it, and how results are written back to memory.

  1. Agent Decides on Tool Call
    The LLM, given the current prompt (Working Memory + skill context), decides it needs to perform an action — for example: read a specific file, search for a function definition, or run a shell command. It outputs a structured ToolCall object specifying the tool name and arguments. The Agent Loop validates this against the registered skill manifest before proceeding.
  2. Skill Loader Resolves Tool
    The Skill Loader looks up the tool name in the registry. If it's a built-in skill (e.g., read_file, search_code), the local executor is used. If it's an MCP tool, the MCP Client is invoked to route the call to the external server. The tool's input schema is validated against the LLM-supplied arguments.
  3. Tool Executor Invokes the Tool
    The Tool Executor dispatches the call. For local file/shell operations, it uses Python's subprocess with a configurable timeout (default 30s) and memory limit. For sandboxed tools, it spawns an isolated subprocess or Docker container. For MCP tools, it sends a JSON-RPC request to the MCP server. The Observer logs the invocation start event.
  4. Result Serialized as ToolResult
    Regardless of the underlying runtime, the result is wrapped in a normalized ToolResult schema: stdout (string), stderr (string), exit_code (int), duration_ms (float), and optional error (string). The Tool Executor truncates output at a configurable limit (default 1 MB) to prevent memory exhaustion.
  5. Result Written Back to Working Memory
    The ToolResult is appended to the session's Working Memory under a keyed entry (e.g., tool_call_results[{uuid}]). This entry is accessible to the LLM in the next reasoning iteration. The full tool call history is also persisted to the Checkpoint snapshot for potential resume.
  6. Long-term Memory Updated (If Relevant)
    If the tool result contains a persistent entity (e.g., a newly discovered entry point or dependency version), the Long-term Memory is updated with a new JSON fact keyed by the entity type and identifier. This enables future analysis sessions to leverage previous findings without re-discovering the same information.
  7. Agent Loop Continues or Converges
    The Agent Loop feeds the ToolResult back into the LLM context. The LLM decides whether the result resolves the current reasoning gap (in which case it moves to the next reasoning step) or whether additional tool calls are needed. The Loop Controller checks for convergence: stable findings across two consecutive iterations. If converged or max_iter reached, the loop terminates.
3

Pipeline 3

Evidence Storage Flow

How every LLM conclusion transitions from a candidate finding to a verified, stored evidence unit — or gets flagged and excluded.

  1. LLM Generates a Finding Candidate
    During reasoning, the LLM produces a conclusion — for example: "The login() function at auth.py:42 has no authorization check before returning user data." This conclusion is a candidate finding. It exists only in Working Memory at this stage. The Agent Loop routes it to the MEU enforcement path for validation.
  2. Schema Validator Checks for MEU
    The Schema Validator inspects the candidate finding. A valid MEU requires all three fields: file path (string, must exist and be readable), line range (two integers, must be within file bounds), and code snippet (string, must match the actual source lines at the specified range). The validator reads the file to cross-check the snippet. If any field is missing, malformed, or unverified, the finding is rejected.
  3. MEU Valid — Evidence Store Persisted
    A valid finding is wrapped in an EvidenceUnit and inserted into the Evidence Store (SQLite). The row includes: finding_id (UUID), task_id, skill (origin skill), severity, title, description, file_path, line_start, line_end, snippet, created_at, and verified (always true). The SQLite write is transactional — a failure rolls back cleanly without corrupting existing records.
  4. MEU Invalid — Flagged as UNVERIFIED
    A finding without a valid MEU is stored in a separate unverified_findings table with the same structure, but with verified set to false and a rejection_reason field explaining why (e.g., "file not found", "snippet mismatch", "no MEU provided"). These entries are never included in the final report but are kept for audit and evaluation purposes.
  5. RAG Store Indexed
    For every verified EvidenceUnit, the report_compiler chunks the finding text (title + description + snippet) and inserts it into the RAG Store (SQLite FTS5) with an FTS5 INSERT. The chunk is associated with the task_id, finding_id, and severity. This enables future natural-language queries like "SQL injection in authentication module" to retrieve relevant historical findings with source citations.
  6. Long-term Memory Updated with Entity Fact
    After a finding is verified and stored, a summary fact is written to Long-term Memory. For a confirmed vulnerability, this might be: {"type": "vuln", "file": "auth.py", "line": 42, "severity": "high", "cve": null, "task_id": "uuid"}. This persists the entity across sessions so that future analysis of the same codebase can start from existing knowledge rather than re-discovering the same vulnerability.
  7. Report Compiler Includes Only Verified Findings
    When report_compiler runs, it queries the Evidence Store with WHERE verified = true AND task_id = :task_id. Unverified findings are excluded. The report therefore contains only conclusions that pass the MEU enforcement gate — every claim is traceable to a file path, line range, and code snippet.
End-to-end guarantee: Every line in the generated report traces back to a verified EvidenceUnit in the SQLite Evidence Store. If the file changes, the snippet no longer matches — and the next run of odin analyze will flag the stale finding as STALE_EVIDENCE, requiring re-verification.

Summary

Three pipelines, one goal

All three pipelines share the same memory layer and converge at the final report.

Pipeline Trigger Key Modules Output
Analysis Request odin analyze / POST /tasks FastAPI → Workflow Orchestrator → Pipeline Executor → Report Compiler .odin/audit/report-{date}.md
Tool Calling LLM reasoning decision Agent Loop → Skill Loader → Tool Executor → Working Memory ToolResult written to session memory
Evidence Storage LLM conclusion generated LLM → Schema Validator → Evidence Store + RAG Store SQLite row in evidence_units table
