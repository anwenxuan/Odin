Tool Executor Layer

The Tool layer is the foundation of Odin's capabilities — it gives the LLM real "hands" to interact with the codebase. Every tool follows the Tool Protocol interface and returns a ToolResult, regardless of internal success or failure. This uniform interface makes the system robust and debuggable.

Tools are exposed to the LLM in OpenAI Function Calling format, which means the LLM can request specific tools with arguments just like calling an API function.

Core Interfaces

ToolResult

The universal return type for all tool executions.

tools/base.py
@dataclass
class ToolResult:
    success: bool
    output: str = ""       # Human-readable output
    error: str | None = None
    metadata: dict = field(default_factory=dict)

    @classmethod
    def ok(cls, output: str, **meta):
        return cls(success=True, output=output, **meta)

    @classmethod
    def err(cls, error: str, **meta):
        return cls(success=False, error=error, **meta)

ToolContext

Runtime context passed to every tool execution.

tools/base.py
@dataclass
class ToolContext:
    repo_path: Path | None = None   # Repository root
    session_id: str = ""
    run_id: str = ""
    skill_id: str = ""
    custom: dict = field(default_factory=dict)

    def resolve_path(self, rel_path: str) -> Path | None:
        # Prevents path traversal — rejects paths
        # that escape the repo root
        p = (self.repo_path / rel_path).resolve()
        p.relative_to(self.repo_path.resolve())
        return p

read_file

Reads the content of a file, optionally limited to a specific line range. This is the most frequently used tool — the LLM reads source files to extract evidence.

tools/builtin/read_file.py
@tool
class ReadFileTool:
    name = "read_file"
    description = (
        "Read the content of a file. "
        "Use line_start and line_end to read specific sections. "
        "Always use this before analyzing any code."
    )
    input_schema = {
        "type": "object",
        "properties": {
            "path": {"type": "string", "description": "File path (relative or absolute)"},
            "line_start": {"type": "integer", "description": "Start line (1-indexed, inclusive)"},
            "line_end": {"type": "integer", "description": "End line (1-indexed, inclusive)"},
        },
        "required": ["path"]
    }

    def execute(self, args: dict, ctx: ToolContext) -> ToolResult:
        resolved = ctx.resolve_path(args["path"])
        if not resolved:
            return ToolResult.err("Path traversal blocked: cannot read outside repo")
        # ... file reading logic ...
ℹ

Path safety: ctx.resolve_path() uses Path.relative_to() to ensure the resolved path never escapes the repository root. Attempts to read ../../etc/passwd are blocked and return an error.

search_code

Performs regex search across all files in the repository with configurable context lines. The LLM uses this to find functions, classes, patterns, and specific code constructs.

tools/builtin/search_code.py

class SearchCodeTool:

        "Returns matching lines with surrounding context. "
        "Useful for finding function definitions, imports, security patterns."
    )
    input_schema = {
        "type": "object",
        "properties": {
            "pattern": {
                "type": "string",
                "description": "Regex pattern to search for"
            },
            "file_pattern": {
                "type": "string",
                "description": "File glob pattern (e.g. '*.py', 'src/**/*.js')"
            },
            "context": {
                "type": "integer",
                "description": "Number of context lines before/after matches (default: 3)"
            },
            "case_sensitive": {"type": "boolean", "default": False},

        "required": ["pattern"]
    }

run_shell

Executes safe shell commands from a strict whitelist. This tool is designed to allow the LLM to use standard Unix utilities without the risk of arbitrary command execution.

⚠

Security: Strict command whitelist. Only pre-approved commands can run. The LLM cannot execute arbitrary shell commands, scripts, or pipes. This is a critical security boundary.

Approved commands:

Git

git, git-log, git-diff, git-show, git-blame

File ops

ls, cat, head, tail, wc, stat, file

Search

grep, find, rg, cut, sort, uniq

Analysis

md5sum, sha256sum, strings, objdump, nm

Adding Custom Tools

Add new tools using the @tool decorator. The tool is automatically discovered and registered by ToolExecutor.auto_load_builtin().

tools/builtin/my_custom_tool.py
from tools.base import tool, ToolResult, ToolContext

@tool
class MyCustomTool:
    """
    Custom tool for analyzing specific file types.
    Add @tool decorator to auto-register.
    """
    name = "my_custom_tool"
    description = (
        "A brief description of what this tool does. "
        "Be specific about inputs and when to use it."
    )

    input_schema = {
        "type": "object",
        "properties": {
            "path": {
                "type": "string",
                "description": "Path to the target file"
            },
            "option": {
                "type": "string",
                "description": "An optional configuration",
                "enum": ["option_a", "option_b"]
            }
        },
        "required": ["path"]
    }

    def execute(self, args: dict, ctx: ToolContext) -> ToolResult:
        path = args["path"]
        option = args.get("option", "default")

        # Use ctx.resolve_path() for security
        resolved = ctx.resolve_path(path)
        if not resolved:
            return ToolResult.err("Invalid path")

        try:
            # Your tool logic here
            result = do_something(resolved, option)
            return ToolResult.ok(
                output=f"Analysis complete: {result}",
                metadata={"file": str(resolved), "option": option}
            )
        except Exception as exc:
            return ToolResult.err(f"Error: {exc}")
ℹ

Custom tools are automatically added to the LLM's function calling list. The LLM will decide when to use your tool based on its description — make descriptions clear and actionable.

Security Model

Odin implements multiple layers of security to prevent misuse:

Path Traversal Prevention

All file operations use ctx.resolve_path() which rejects any path that escapes the repository root.

Command Whitelist

run_shell only executes pre-approved commands. The LLM cannot run arbitrary shell code.

Tool Call Logging

Every tool execution is logged with session ID, tool name, arguments, and result — enabling full audit trails.
