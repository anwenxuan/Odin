Guide / Troubleshooting

Troubleshooting

Diagnose and resolve common issues with Odin. Find the problem, understand the cause, and apply the fix.

API key not set / authentication error

Error

odin.errors.AuthError: No API key configured for provider "openai". Set OPENAI_API_KEY or configure provider in odin.yaml.

Cause

Odin requires a valid API key for any LLM provider except Mock mode. If the OPENAI_API_KEY (or equivalent) environment variable is missing, or if the key in odin.yaml is invalid or expired, authentication fails at the first API call.

Solution

Step 1: Verify the environment variable is set:

bash 
echo $OPENAI_API_KEY   # Should print your key, not be empty

Step 2: Set the variable (replace with your actual key):

bash 
export OPENAI_API_KEY=sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

Step 3: Or configure it in odin.yaml:

yaml 
# odin.yaml
provider: openai
model: gpt-4o-mini
api_key: "sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"   # Or use env var: env:OPENAI_API_KEY
Never commit API keys to source control.
Use environment variables or a .env file (add .env to .gitignore). In CI/CD, pass keys via secret variables.

Analysis request timed out

Error

odin.errors.TimeoutError: Analysis request timed out after 300s. Target: src/auth/, workflow: deep-audit.

Cause

Timeouts occur when the LLM takes longer than the configured threshold to respond. Common triggers include:

  • Very large repositories with thousands of files
  • Complex multi-step workflows with many sequential skill calls
  • LLM API rate limiting causing retries
  • Network latency when using remote providers (e.g., OpenAI from a region with high ping)

Solution

Increase the timeout in odin.yaml:

yaml 
analysis:
  timeout: 600    # seconds, default is 300
  include: ["src/**/*.py"]   # narrow scope to reduce processing time

Or pass it via CLI for a one-off run:

bash 
odin analyze ./src --timeout 600

If timeouts persist, try using a faster model (gpt-4o-mini or claude-3-haiku) or reduce the analysis scope.

MCP server connection failed

Error

odin.mcp.ConnectionError: Failed to connect to MCP server at localhost:8080. Is the server running?

Cause

The Model Context Protocol (MCP) server that Odin connects to for extended tooling (file system access, git operations, etc.) is either not running or unreachable at the configured address.

Solution

Step 1: Confirm the MCP server is running:

bash 
curl http://localhost:8080/health
# Expected: {"status": "ok"}

Step 2: Start the MCP server if it's not running:

bash 
# Start the bundled MCP server in the background
odin mcp serve &
# Or start a custom MCP server
python -m odin.mcp.server --port 8080

Step 3: Check the configured address in odin.yaml:

yaml 
mcp:
  enabled: true
  server_url: "http://localhost:8080"
  timeout: 30

RAG vector store permission denied

Error

PermissionError: [Errno 13] Permission denied: '/root/.odin/rag/vector_store.db'

Cause

Odin stores RAG vector embeddings in ~/.odin/rag/. If that directory was created by a different user (e.g., root during a Docker run), the current user may lack write permissions.

Solution

Fix ownership of the RAG directory:

bash 
sudo chown -R $(whoami) ~/.odin/rag/

Or use a custom path that your user owns:

yaml 
rag:
  storage_path: "./odin-rag/"   # relative to project root

DAG execution hangs / task never completes

Error

odin.dag.DAGStuckError: Node "skill:dependency-graph" has been PENDING for 420s. No progress detected.

Cause

A DAG node gets stuck when a skill task is waiting indefinitely — usually because its upstream dependencies never resolved, or the LLM call returned an unparseable response that the skill handler couldn't process.

Solution

Step 1: Check the DAG execution log for the stuck node:

bash 
cat odin-output/$TARGET/dag/execution.json | grep -A 5 "PENDING"

Step 2: Enable strict dependency validation to catch circular or missing dependencies early:

yaml 
dag:
  strict_mode: true   # Fail fast on circular deps or missing edges
  node_timeout: 120   # Per-node timeout in seconds

Step 3: Use a lighter workflow if the full DAG is too complex:

bash 
# Use the quick-audit workflow instead of deep-audit
odin analyze ./src --workflow quick-audit

Out of memory / OOM on large repositories

Error

MemoryError: Unable to allocate RAG chunk buffer. Repository size: 2.4 GB. Available RAM: 1.8 GB.

Cause

Large repositories with thousands of files generate massive RAG vector embeddings. If the total chunked text exceeds available RAM, the process is killed by the OS OOM killer or raises a MemoryError. This is especially common when processing entire monorepos without scope limits.

Solution

Step 1: Limit the files being processed:

yaml 
analysis:
  include: ["src/**/*.py"]
  exclude: ["tests/**", "node_modules/**", "vendor/**", "dist/**"]

Step 2: Increase RAG chunk overlap and reduce batch size to lower peak memory:

yaml 
rag:
  chunk_size: 512       # tokens per chunk (default: 1024)
  chunk_overlap: 64    # overlap between chunks
  batch_size: 50        # embeddings per API call (default: 100)

Step 3: For very large repos, process in batches by directory:

bash 
# Analyze subdirectories one at a time
odin analyze ./src/auth --output ./odin-output/auth
odin analyze ./src/api --output ./odin-output/api
odin analyze ./src/core --output ./odin-output/core
Still stuck?
Run odin diag to generate a full system diagnostic report and share it in a GitHub Issue.
