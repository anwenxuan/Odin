Guide / Advanced Usage

Advanced Usage

Customize every layer of Odin's pipeline — from bespoke Skills to GitHub Actions integration.

Custom Skill Development

Odin's Skill interface is deliberately minimal. Implement BaseSkill, define your schemas, and the Pipeline handles everything else — validation, parallel execution, and MEU collection.

Skill Structure

A Skill is a Python class with four key attributes and one required method:

  • name — unique identifier used in Workflow YAML
  • description — human-readable summary shown in odin skills list
  • input_schema — JSON Schema for what the Skill receives
  • output_schema — JSON Schema for what the Skill emits
  • execute(ctx, **kwargs) — the main logic, returns a list of MEU dicts
python 
# ~/.odin/skills/bandit_audit.py
from typing import Any
from odin.skills.base import BaseSkill
from odin.core.context import SkillContext

BLOCKED_LICENSES = ["GPL-3.0", "AGPL-3.0", "LGPL-3.0"]

class BanditAuditSkill(BaseSkill):
    name = "bandit_audit"
    description = (
        "Run Bandit static analysis on Python files. "
        "Emits MEUs for each issue found with CWE mapping."
    )
    input_schema = {
        "type": "object",
        "properties": {
            "target": {"type": "string"},
            "severity": {"type": "string", "enum": ["low", "medium", "high"]},
        },
        "required": ["target"],
    }
    output_schema = {"type": "array", "items": {"type": "object"}}

    async def execute(self, ctx: SkillContext, target: str, severity: str = "medium") -> list[dict[str, Any]]:
        result = ctx.call_tool(
            "run_command",
            cmd=["bandit", "-r", target, "-f", "json", "-ll"]
        )

        findings = []
        for issue in result.get("results", []):
            if issue["issue_severity"] != severity:
                continue
            findings.append(ctx.emit_meu(
                file=issue["filename"],
                line_start=issue["line_number"],
                line_end=issue["line_number"],
                snippet=issue["code"],
                confidence="verified",
                tags=["bandit", issue["issue_cwe"]],
            ))
        return findings

Place the file in ~/.odin/skills/ (or .odin/skills/ within a project). Odin auto-discovers and registers it on next run. No restart needed.

SkillContext gives you access to ctx.call_tool() for raw tool calls, ctx.emit_meu() for creating evidence units, and ctx.working_memory for sharing state across Skills in the same workflow.

Custom Workflow YAML

Workflow YAML files live in .odin/workflows/. Define your own pipeline to match your team's security posture and domain focus.

YAML Structure Reference

yaml 
# .odin/workflows/custom_secure_audit.yaml
# Full-featured workflow with MCP, RAG context, and layered execution

name: custom_secure_audit
description: "Security audit with Bandit, secrets, and CVE cross-reference"

# Optional: enrich agent context from RAG before analysis
rag_context:
  query: "previous vulnerabilities in {repo_name}"
  top_k: 5

# Optional: MCP tools available in this workflow
mcp:
  - github    # access repo metadata, issues, PR history
  - cve_db    # cross-reference dependency versions against CVE feed

# DAG layers — Skills within a layer run in parallel
layers:

  # Layer 1: Surface analysis — single task, no dependencies
  - id: surface
    skills:
      - repo_discovery
      - detect_language          # custom built-in Skill

  # Layer 2: Dependency + secret scanning — runs after surface completes
  - id: dependencies
    depends_on: [surface]
    skills:
      - dependency_audit
      - secret_detection
      - bandit_audit             # your custom Skill from ~/.odin/skills/

  # Layer 3: Code structure analysis — runs after dependencies
  - id: structure
    depends_on: [dependencies]
    skills:
      - entrypoints
      - call_graph
      - cfg_builder              # only for complex files (>50 LOC)

  # Layer 4: Security deep-dive — depends on structure layer
  - id: security
    depends_on: [structure]
    skills:
      - auth_guard_analysis
      - taint_analysis
      - vulnerability_scan       # uses CVE context from MCP

  # Layer 5: Finalize — depends on security, always runs last
  - id: finalize
    depends_on: [security]
    skills:
      - report_gen

# Override default confidence threshold for report inclusion
report:
  min_confidence: medium
  format: markdown
  output: ".odin/audit/{date}-report.md"

Run a custom workflow with:

bash 
$ odin analyze ./my-project \
    --workflow custom_secure_audit \
    --provider openai

MCP Tool Integration

Odin ships a built-in MCP client that can connect to any MCP-compatible server. Use it to enrich analysis with live data from GitHub, CVE databases, Jira, and more.

Configuration

Add MCP servers to your odin.yaml:

yaml 
# odin.yaml
mcp:
  github:
    # Authenticate via GITHUB_TOKEN environment variable
    enabled: true
    tools:
      - get_repo_metadata
      - list_issues
      - get_pull_requests

  cve_db:
    enabled: true
    url: "https://services.nvd.nist.gov/rest/json/cves/2.0"
    api_key: "${NVD_API_KEY}"    # read from env
    tools:
      - cve_lookup
      - cwe_details

  jira:
    enabled: false             # disabled by default
    url: "https://your-org.atlassian.net"
    token: "${JIRA_TOKEN}"
    email: "${JIRA_EMAIL}"

Within a Skill, access MCP tools through ctx.mcp_tool():

python 
# In your Skill's execute() method:
# Cross-reference detected dependency versions against CVEs
cve_results = ctx.mcp_tool(
    "cve_db",
    "cve_lookup",
    package=dep.name,
    version=dep.version,
)

# Check if this repo has known CVEs mentioned in issues
github_issues = ctx.mcp_tool(
    "github",
    "list_issues",
    repo=ctx.target_repo,
    labels=["security", "cve"],
)

Batch Analysis

Analyze multiple repositories in a single command using a targets file or a directory scan.

bash 
# Analyze a directory of repositories in parallel (4 at a time)
$ odin analyze ./repos/ \
    --workflow vulnerability_research \
    --provider openai \
    --parallel 4 \
    --report-dir ./audit-results/

# Analyze a list of targets from a file (one repo URL per line)
$ odin analyze --targets ./repos.txt \
    --workflow dependency_audit \
    --provider anthropic \
    --parallel 8 \
    --report-dir ./audit-results/

# Run a dry-run that only lists what would be analyzed
$ odin analyze ./repos/ \
    --dry-run \
    --workflow vulnerability_research
Rate limiting
When running batch analysis with --parallel, respect provider rate limits. Use --parallel 1 for OpenAI and --parallel 4 for Anthropic by default. Adjust with --rpm to set a custom requests-per-minute cap.

CI/CD Integration

Run Odin as a GitHub Actions step in your CI pipeline. It can gate merges on critical findings or just generate reports.

GitHub Actions

yaml 
# .github/workflows/security-audit.yml
name: Security Audit

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]
  schedule:
    # Weekly full audit on the main branch
    - cron: "0 3 * * 1"

jobs:
  odin-audit:
    name: Odin Security Audit
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install Odin
        run: |
          pip install -e .

      - name: Run Odin audit
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          odin analyze . \
            --workflow vulnerability_research \
            --provider openai \
            --output .odin/audit/report.json \
            --fail-on high

      - name: Upload report artifact
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: odin-audit-report
          path: .odin/audit/

      - name: Post comment on PR
        if: github.event_name == 'pull_request'
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `## Odin Security Report\n\nAudit complete. Check the artifact for full findings.`
            })
Fail gates
Use --fail-on high (or --fail-on medium) to make the CI job non-zero if Odin finds findings at or above that severity. This is ideal for gating main branch merges.
