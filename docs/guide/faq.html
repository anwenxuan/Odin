Guide / FAQ

Frequently Asked Questions

Everything you need to know about using Odin for systematic vulnerability research.

Do I need an API key to use Odin?

No — Odin ships with a built-in Mock mode that requires no external API key.

When you run Odin without configuring an LLM provider, it automatically falls back to Mock mode. In this mode, Odin simulates the full analysis pipeline using deterministic, rule-based responses. This is ideal for:

  • Evaluating Odin's workflow and DAG execution behavior
  • Testing skill orchestration and MEU enforcement locally
  • Learning the Odin CLI and configuration system
  • CI/CD environments where external API calls are undesirable
Real analysis requires an API key.
To perform actual vulnerability research on real codebases, configure a supported LLM provider (OpenAI, Anthropic, Azure OpenAI, or Ollama) in odin.yaml.

Which LLM providers does Odin support?

Odin is provider-agnostic and works with any OpenAI-compatible endpoint.

OpenAI

GPT-4o, GPT-4o-mini, GPT-4-Turbo. Set provider: openai and model: gpt-4o.

Anthropic

Claude 3.5 Sonnet, Claude 3 Opus. Set provider: anthropic and model: claude-3-5-sonnet.

Azure OpenAI

Deployments behind Azure's API. Set provider: azure with your endpoint and API version.

Ollama

Local models (Llama 3, Mistral, Qwen). Set provider: ollama with your local endpoint URL.

Function calling required.
All providers must support OpenAI-compatible function calling (tool use) for Odin to invoke skills and workflow steps.

Where are analysis results stored?

Results are persisted locally in structured formats, with full control over storage location.

By default, Odin stores all outputs under ./odin-output/:

output 
odin-output/
├── <target-name>/
│   ├── analysis/
│   │   ├── <timestamp>.json      # Full analysis report
│   │   ├── evidence/             # MEU evidence files
│   │   └── skills/               # Per-skill outputs
│   ├── dag/
│   │   └── execution.json       # DAG execution trace
│   └── logs/
│       └── odin.log              # Execution logs

Customize the output directory via the odin.yaml output.path field or the --output CLI flag. RAG vector stores are kept in ~/.odin/rag/ by default.

How does Odin handle private repositories?

Odin clones repositories on-demand and never uploads code to external servers (unless you configure a cloud LLM).

For private repos, authenticate before running analysis:

bash 
# HTTPS with personal access token
git clone https://github.com/YOUR_ORG/private-repo.git

# SSH (preferred)
git clone git@github.com:YOUR_ORG/private-repo.git

Alternatively, pass a GitHub token via the GITHUB_TOKEN environment variable — Odin will inject it automatically when cloning HTTPS URLs:

bash 
export GITHUB_TOKEN=ghp_xxxxxxxxxxxx
odin analyze https://github.com/YOUR_ORG/private-repo

The analysis is running slowly. What can I do?

Slowness is usually caused by model latency, repository size, or suboptimal workflow configuration.

Here are the most effective optimizations, in order of impact:

  1. Use a faster model. GPT-4o-mini and Claude 3.5 Haiku are significantly cheaper and faster than GPT-4o or Claude 3 Opus, with comparable accuracy for routine vulnerability patterns.
  2. Limit the target scope. Run analysis on a specific subdirectory or file pattern rather than an entire monorepo. Use odin.yaml to configure include/exclude patterns:
yaml 
# odin.yaml — focus on Python files only
analysis:
  include: ["**/*.py"]
  exclude: ["tests/**", "venv/**", "node_modules/**"]
  1. Enable DAG parallel execution. Skills that are independent (different files or modules) run in parallel by default. Verify your workflow is configured with correct dependency edges in the DAG.
  2. Reduce RAG chunk size. Smaller chunks (512 tokens vs 2048) mean fewer retrieval calls. Adjust in odin.yaml under rag.chunk_size.
  3. Cache repeated analyses. Re-running the same target skips unchanged files when analysis.cache: true is enabled.

What does "MEU verification failed" mean?

The Minimum Evidence Unit (MEU) for a finding did not meet the minimum evidence threshold.

MEU is a core Odin concept — every vulnerability claim must be backed by a structured evidence unit containing:

  • file — the source file path
  • line_range — exact or approximate line numbers
  • snippet — the relevant code excerpt
  • confidence — a score from 0.0 to 1.0
  • rationale — a natural-language explanation of why this is a finding

A MEU verification fails when any of these fields is missing, empty, or when confidence falls below the threshold set in odin.yaml (default: 0.6). Failed MEUs are logged as warnings and excluded from the final report.

bash 
# Adjust the MEU confidence threshold
meu:
  min_confidence: 0.5   # Lower = more permissive, more false positives
  require_snippet: true  # Enforce that snippet is never empty

How do I contribute a new skill?

Skills are defined as structured YAML files with prompts, triggers, and evidence schemas.

Each skill lives in ~/.odin/skills/ or ./skills/ as a YAML file. Here is the minimal structure:

yaml 
name: detect-sql-injection
description: Detects SQL injection vulnerabilities in string concatenations
language: python
trigger: "**.py"
priority: p0

prompt: |
  Review the provided Python file for SQL injection risks.
  ...

meu_schema:
  required_fields: [file, line_range, snippet, confidence, rationale]
  min_confidence: 0.7

output:
  format: json
  path: "skills/sql-injection-findings.json"

Submit your skill as a pull request to the Odin community skills repository. See the Core Concepts guide for the full skill schema reference.

How do I integrate Odin with GitHub Actions?

A dedicated GitHub Action automates vulnerability scanning on every pull request.

Add this to your workflow file (.github/workflows/security-scan.yml):

yaml 
name: Odin Security Scan

on:
  pull_request:
    paths:
      - "**.py"
      - "**.js"

jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: anwenxuan/odin-action@v0.2
        with:
          provider: openai
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}

The action posts findings as pull request comments and fails the build if MEU-validated high-severity vulnerabilities are detected.
