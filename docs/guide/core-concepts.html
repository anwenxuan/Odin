Guide / Core Concepts

Core Concepts

The foundational ideas that power Odin's evidence-backed, parallel, autonomous analysis.

MEU — Minimum Evidence Unit

Every conclusion Odin produces must be anchored to a concrete code location. A MEU is that anchor — the smallest verifiable unit of evidence.

Without evidence, there is no finding. Odin enforces this at the system level: the Evidence Validator rejects any claim that cannot be traced back to a specific file, line range, and snippet. This design eliminates the hallucinations that plague general-purpose AI code assistants.

A MEU is structured as:

  • File path — absolute or relative path within the target repository
  • Line range — start and end line numbers
  • Snippet — the exact code text (byte-for-byte match)
  • Context — surrounding code for human verification
  • Confidence — low / medium / high / verified
json 
{
  "type": "MEU",
  "id": "meu_003",
  "file": "src/auth.py",
  "line_start": 42,
  "line_end": 58,
  "snippet": "def verify_token(token):\n    # WARNING: no expiry check\n    return jwt.decode(token, options={'verify_signature': False})",
  "confidence": "high",
  "skill": "auth_guard_analysis",
  "timestamp": "2026-03-26T10:24:00Z"
}
Why it matters
MEU enforcement means every vulnerability claim in your final report links directly to source code. Security teams can audit findings without re-running the tool — the evidence is self-contained.

Skills

A Skill is an encapsulated analysis capability — like a function in a library. Each Skill handles one specific task and can be composed into any Workflow.

Skills are the atomic units of Odin's execution layer. Each Skill has a defined input schema, a prompt template, tool definitions, and an output schema validated against the MEU model. Twelve Skills ship built-in:

Skill Name Category Description
repo_discovery Surface Analysis Clone or scan a target repository, detect language, enumerate files and directories
entrypoints Surface Analysis Identify public APIs, CLI entry points, web handlers, and RPC endpoints
call_graph Structure Analysis Map function-to-function call chains and build a directed graph of dependencies
auth_guard_analysis Security Analysis Detect missing authentication, authorization checks, and privilege escalation paths
vulnerability_scan Security Analysis Cross-reference call chains against known vulnerability patterns (CWE mapping)
dependency_audit Dependency Analysis Analyze package manifests, detect known vulnerable or outdated dependencies
taint_analysis Security Analysis Trace user-controlled input through sinks (SQL, command execution, file I/O)
data_flow Structure Analysis Map how data moves through the codebase — from source to persistence or network
secret_detection Security Analysis Scan for hardcoded credentials, API keys, tokens, and secrets in source or config
cfg_builder Structure Analysis Build control flow graphs for complex functions to expose unreachable branches
exploit_gen Exploitation Generate proof-of-concept payloads that exercise identified vulnerability paths
report_gen Output Aggregate MEUs into a structured Markdown/JSON report with severity scoring

Custom Skills can be registered via the ~/.odin/skills/ directory. Each is a Python class that inherits from odin.skills.base.BaseSkill.

python 
from odin.skills.base import BaseSkill

class LicenseAuditSkill(BaseSkill):
    name = "license_audit"
    description = "Detect incompatible or missing licenses in dependencies"
    input_schema = {"type": "object", "properties": {}}}
    output_schema = {"type": "array"}

    async def execute(self, ctx, target):
        deps = ctx.call_tool("read_manifest", path=target)
        findings = []
        for dep in deps:
            if dep.license in BLOCKED_LICENSES:
                findings.append(ctx.emit_meu(
                    file=dep.manifest,
                    snippet=dep.license_line,
                    confidence="verified"
                ))
        return findings

Workflows

A Workflow is an ordered composition of Skills, defined in YAML. It describes the analysis pipeline — which Skills run, in what order, and with what dependencies.

Workflows turn Skills into reproducible research processes. A single YAML file defines the entire analysis pipeline, making it easy to version-control, share, and audit your methodology.

yaml 
# .odin/workflows/vulnerability_research.yaml
name: vulnerability_research
description: "Full pipeline: surface → graph → security → report"

layers:
  - id: layer_1
    skills:
      - repo_discovery

  - id: layer_2
    depends_on: [layer_1]
    skills:
      - entrypoints
      - call_graph

  - id: layer_3
    depends_on: [layer_2]
    skills:
      - auth_guard_analysis
      - taint_analysis
      - secret_detection

  - id: layer_4
    depends_on: [layer_3]
    skills:
      - vulnerability_scan
      - report_gen

The depends_on field declares layer boundaries. Skills within the same layer execute in parallel (via ThreadPoolExecutor). Skills across layers execute sequentially. This is the DAG-layer model.

Evidence Model

The Evidence Model governs how discoveries are recorded, validated, stored, and retrieved throughout the analysis lifecycle.

Every Skill emits findings as MEUs. Before a finding enters the Evidence Store, it passes through the Evidence Validator:

  1. Schema check — Does the finding match the output schema for the skill that produced it?
  2. File existence — Does the referenced file actually exist in the cloned repository?
  3. Snippet verification — Does the exact snippet appear at the claimed line range?
  4. Context enrichment — Two lines of surrounding code are appended automatically for human review
  5. Confidence scoring — Each finding is tagged low / medium / high / verified
Verified findings
Findings with confidence: "verified" have passed all three validation checks. These are the findings included in the final report by default.

Validated MEUs are persisted to .odin/audit/evidence.jsonl (one JSON object per line) and indexed in the SQLite FTS5 RAG store for natural-language querying via the /skills or /workflows API endpoints.

DAG Execution

Odin models the analysis pipeline as a Directed Acyclic Graph (DAG). Independent Skills execute in parallel; dependent Skills wait until their prerequisites complete.

Traditional sequential pipelines execute Skill A, then B, then C — even when B and C don't depend on each other. Odin computes the dependency graph from the Workflow YAML and identifies "layers" of parallelizable work.

In this example, Layer 2 has three Skills that all depend only on Layer 1's output. They execute in parallel. Layer 3 then runs in parallel. Only Layer 4 (report generation) requires all prior layers to complete. An 8-step sequential pipeline completes in 4 layers.

Agent Loop

The Agent Loop is the cognitive engine of Odin — a cyclical process of reasoning, tool use, and observation that drives the analysis forward.

Built on an LLM adapter, the Agent Loop iterates until either the workflow is complete, a maximum iteration count is reached, or a stopping condition is triggered. Each iteration follows a strict sequence:

  1. LLM Call — The Agent sends the current state (observation summary, memory context, tool list) to the LLM and requests the next action.
  2. Tool Execution — The action (a Skill invocation or a raw tool call) is executed. Tool results are captured as structured output.
  3. Observation — The tool results are summarized as natural-language observations and added to the working memory.
  4. Loop — The updated state is fed back to step 1. The loop continues until the workflow completes or a stop condition fires.
text 
Iteration 1
  Agent: "repo_discovery completed. 14 files found."
  Action: invoke skill[call_graph] with files=[...14]
  Tool: call_graph.execute() → 12 call chains
  Observation: "Found 12 chains. auth module has 3 uncovered paths."

Iteration 2
  Agent: "Focus on auth module chains 4, 7, 9 — no guards detected."
  Action: invoke skill[auth_guard_analysis] with chains=[4, 7, 9]
  Tool: auth_guard_analysis.execute() → 2 MEUs emitted
  Observation: "2 MEUs: missing role check at admin.py:142, no token expiry at auth.py:58."

Iteration 3
  Agent: "Both findings verified. Proceeding to report_gen."
  Action: invoke skill[report_gen] with evidence=[meu_001, meu_002]
  Tool: report_gen.execute() → report.md written
  Observation: "Report generated with 2 verified findings."
  → Loop terminates: workflow complete.

The Agent Loop is instrumented with checkpointing, retry logic, and error handling. If a tool call fails, the Agent receives the error as an observation and decides whether to retry with different arguments or skip the step.
