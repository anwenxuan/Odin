What is a Workflow?

A Workflow is a directed acyclic graph (DAG) of analysis steps, defined in YAML. Each step references a Skill and defines its inputs through variable resolution. The PipelineExecutor loads a workflow definition, groups steps into DAG layers, and executes them with maximum parallelism.

ℹ

Workflows live in workflows/ as YAML files. You can add custom workflows by placing a new YAML file in that directory — no code changes required.

Workflow YAML Structure

workflows/vulnerability_research.yaml
id: vulnerability_research
name: Full Vulnerability Research
version: "1.0"
description: >
  Comprehensive 11-step security analysis workflow.
  Covers entry points, call graph, auth logic, input flows,
  sink detection, attack surfaces, and PoC generation.

steps:
  - id: repo_discovery
    skill: repo_map
    with_params:
      deep_scan: true

  - id: entrypoints_detection
    skill: entrypoints_detection
    depends_on: [repo_discovery]

  - id: sink_detection
    skill: sink_detection
    depends_on: [repo_discovery]

  - id: dependency_analysis
    skill: dependency_analysis
    depends_on: [repo_discovery]
    # ... continues ...

DAG Layering & Parallel Execution

The PipelineExecutor uses Kahn's algorithm to topologically sort workflow steps into layers. Steps in the same layer have no mutual dependencies and therefore can execute in parallel via ThreadPoolExecutor.

Layering Algorithm

core/pipeline_executor.py
def _group_by_dag_level(self, definition: WorkflowDefinition) -> list[ExecutionLayer]:
    """
    Kahn's algorithm variant — O(V+E) topological sort.

    1. Compute in-degree for each step (number of dependencies)
    2. Find all steps with in-degree 0 → add as next layer
    3. Remove those steps from the graph, update in-degrees
    4. Repeat until all steps are assigned to a layer
    """
    step_map = {s.id: s for s in definition.steps}
    in_degree = {s.id: 0 for s in definition.steps}

    for step in definition.steps:
        for dep in step.depends_on:
            if dep in in_degree:
                in_degree[step.id] += 1

    layers = []
    completed = set()

    while len(completed) < len(definition.steps):
        ready = [s for s in definition.steps
                 if s.id not in completed and in_degree[s.id] == 0]
        layers.append(ExecutionLayer(stage=len(layers)+1,
                                     step_ids=[s.id for s in ready],
                                     steps=ready))
        for s in ready:
            completed.add(s.id)
            for other in definition.steps:
                if s.id in other.depends_on:
                    in_degree[other.id] -= 1

    return layers

Parallel Execution

core/pipeline_executor.py
def _run_layer_parallel(self, layer: ExecutionLayer, ...) -> dict[str, ParallelStepResult]:
    if len(layer.steps) == 1:
        # Single step — run directly
        return {layer.steps[0].id: self._execute_single_step(...)}

    # Multiple steps — run in ThreadPoolExecutor
    with ThreadPoolExecutor(max_workers=min(len(layer.steps), self.max_workers)) as pool:
        futures = {
            step.id: pool.submit(self._execute_single_step, step=step, ...)
            for step in layer.steps
        }
        results = {sid: f.result() for sid, f in futures.items()}
    return results
ℹ

Performance impact: In vulnerability_research, Layer 2 has 3 steps that run concurrently. On a 3-core machine, this layer completes ~3× faster than sequential execution.

Variable Resolution

Workflow step inputs support two variable substitution syntaxes:

SyntaxSourceExample
${inputs.x}Workflow-level inputs"repo_path": "${inputs.repo_path}"
${steps.S.outputs.y}Previous step outputs"module_map": "${steps.repo_discovery.outputs.modules}"

Example: Passing Step Outputs

workflows/vulnerability_research.yaml (excerpt)
steps:
  - id: repo_discovery
    skill: repo_map
    with_params:
      deep_scan: true
      # Output: { modules: [...], tech_stack: {...} }

  - id: entrypoints_detection
    skill: entrypoints_detection
    depends_on: [repo_discovery]
    with_params:
      # Pass the tech stack from repo_discovery to focus the analysis
      detected_stack: "${steps.repo_discovery.outputs.tech_stack}"
      # Also use the workflow-level repo_path input
      repo_path: "${inputs.repo_path}"

  - id: call_graph_trace
    skill: call_graph_trace
    depends_on: [entrypoints_detection, sink_detection]
    with_params:
      # Receive entrypoints list from entrypoints_detection skill
      entrypoints: "${steps.entrypoints_detection.outputs.entrypoints}"
      # Receive sink list from sink_detection skill
      sinks: "${steps.sink_detection.outputs.sinks}"
⚠

Variables are serialized as JSON before substitution. If ${steps.repo_discovery.outputs.modules} resolves to a list ["auth", "api", "core"], it becomes the JSON string "[\"auth\", \"api\", \"core\"] and the tool receives it as a Python list.

Creating Custom Workflows

Add a YAML file under workflows/. The PipelineExecutor automatically discovers and loads it:

workflows/my_custom_analysis.yaml
id: my_custom_analysis
name: My Custom Analysis Workflow
version: "1.0"
description: >
  Custom workflow that focuses on specific areas
  of interest for our codebase.

# Global inputs — can be overridden via CLI --focus-paths
default_inputs:
  focus_paths: []
  max_depth: 5

steps:
  # Step 1: Discovery
  - id: discover
    skill: repo_map
    with_params:
      deep_scan: false

  # Step 2: Entry points (depends on discovery)
  - id: find_entries
    skill: entrypoints_detection
    depends_on: [discover]
    with_params:
      repo_path: "${inputs.repo_path}"

  # Step 3: Call graph (parallel with auth detection)
  - id: trace_calls
    skill: call_graph_trace
    depends_on: [find_entries]
    with_params:
      entrypoints: "${steps.find_entries.outputs.entrypoints}"

  - id: find_auth
    skill: auth_logic_detection
    depends_on: [find_entries]
    with_params:
      focus_paths: "${inputs.focus_paths}"

  # Step 4: Final report (depends on all previous)
  - id: summarize
    skill: report_generation
    depends_on: [trace_calls, find_auth]
    with_params:
      title: "Custom Analysis Report"
      include_diagrams: true

Then run it:

terminal
python odin.py analyze ./my-repo --workflow my_custom_analysis --provider openai
✓

No code changes needed. Simply drop the YAML file into workflows/ and Odin discovers it automatically on the next run.
