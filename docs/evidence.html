Why Evidence Matters

Odin enforces a strict policy: every conclusion must be backed by at least one Minimum Evidence Unit (MEU). This is not a guideline — it is enforced at runtime. If a Skill's output references a claim without a corresponding evidence citation, the framework rejects it.

✓

The principle: "No conclusion without evidence." Every vulnerability hypothesis, architectural insight, or data flow description must cite specific lines of code from the analyzed repository.

This approach provides three guarantees:

  • Traceability — Every finding can be traced back to exact source code lines
  • Reproducibility — Given the same repo and version, the same evidence will be found
  • Verifiability — Human reviewers can verify each evidence citation independently

Minimum Evidence Unit (MEU)

The MinimumEvidenceUnit is the atomic evidence building block. Each MEU captures a precise code citation with full provenance metadata:

memory/models.py
@dataclass
class MinimumEvidenceUnit:
    # Identity
    meu_id: str                        # MEU-abc123def4
    repo: str                          # owner/repo
    commit: str                       # Git commit SHA

    # Code location
    file_path: str                     # Relative to repo root
    symbol: str                        # Function/class/variable name
    line_start: int | None = None     # 1-indexed inclusive
    line_end: int | None = None       # 1-indexed inclusive
    snippet: str                       # Raw source text

    # Semantic annotation
    evidence_type: EvidenceType       # CODE_SNIPPET, CONFIG, etc.
    language: str                      # python, javascript, go...
    framework: str                     # django, express, gin...

    # Relationship (optional)
    relation: CallRelation | None = None  # For call graph evidence

    # Provenance
    extracted_by: str                  # Skill that extracted this
    confidence: float = 1.0           # 0.0 to 1.0
    tags: list[str] = field(default_factory=list)
    custom: dict = field(default_factory=dict)
    timestamp: str                    # UTC ISO timestamp

Evidence Types

MEUs can represent different kinds of evidence, each with different confidence implications:

TypeDescriptionConfidence
CODE_SNIPPET Direct source code excerpt High (1.0)
CONFIG Configuration file content High (0.95)
DEPENDENCY Dependency declaration (package.json, requirements.txt) High (0.95)
SCHEMA Data structure or API schema definition High (0.9)
API_SPEC API specification (OpenAPI, protobuf) High (0.9)
DOC Documentation comment or README excerpt Medium (0.7)
INFERRED Inferred from patterns (requires human verification) Medium (0.6)

Confidence Levels

Conclusions carry a confidence score from 0.0 to 1.0. Odin maps these to human-readable levels:

HIGH
≥ 0.8 Direct evidence from code analysis. High certainty, minimal assumptions.
↓
MEDIUM
0.5–0.8 Some inference required. Reasonable confidence with supporting context.
↓
LOW
0.4–0.5 Significant inference. May require manual verification.
↓
UNCERTAIN
< 0.4 High uncertainty. Requires manual review. Must include uncertainty_note.

ResearchArtifact

A ResearchArtifact wraps the complete output of a single Skill execution, including all extracted MEUs:

memory/models.py
@dataclass
class ResearchArtifact:
    artifact_id: str          # Globally unique
    run_id: str               # WorkflowRun ID
    skill_id: str             # Which skill produced this
    skill_version: str
    kind: ArtifactKind        # MODULE_MAP, ENTRY_POINTS, CALL_GRAPH...

    # The actual output
    content: dict[str, Any]   # Skill's raw JSON output

    # Summary
    summary: str = ""          # One-sentence summary

    # Evidence tracking
    evidence_refs: list[str]  # MEU IDs cited in this artifact
    tags: list[str] = field(default_factory=list)

    created_at: str           # UTC ISO timestamp

Conclusion

A Conclusion is a claim backed by evidence. It is the primary output of Skills that produce analytical findings:

memory/models.py
@dataclass
class Conclusion:
    claim: str                      # The finding statement
    category: str = ""               # architecture | security | data | behavior
    confidence: float = 0.5          # 0.0 to 1.0
    evidence_refs: list[str] = field(default_factory=list)  # MUST be non-empty!
    tags: list[str] = field(default_factory=list)
    uncertainty_note: str = ""      # Required when confidence < 0.4

    def validate(self) -> list[str]:
        """Runtime validation — enforces evidence policy."""
        errors = []
        if not self.claim.strip():
            errors.append("Conclusion claim cannot be empty.")
        if not self.evidence_refs:
            errors.append(
                "Conclusion has no evidence_refs. "
                "Every conclusion must reference at least one MEU."
            )
        if not (0.0 <= self.confidence <= 1.0):
            errors.append(f"Confidence must be in [0,1], got {self.confidence}.")
        if self.confidence < 0.4 and not self.uncertainty_note.strip():
            errors.append(
                f"Confidence={self.confidence} < 0.4 requires uncertainty_note."
            )
        return errors
⚠

Runtime enforcement: The validate() method is called automatically by the framework. If a Conclusion has no evidence_refs, the Skill output is rejected and the step fails. This prevents "hallucinated" conclusions.

RAGStore — Full-Text Search

The RAGStore provides semantic retrieval over historical analysis reports using SQLite FTS5:

rag/store.py
from rag.store import RAGStore

rag = RAGStore(persist_dir="./data/rag")

# Index a completed report
rag.index_report(
    run_id="run_abc123",
    report_text="This codebase contains SQL injection vulnerabilities...",
    repo_url="https://github.com/owner/repo",
    tags=["security", "sql-injection"]
)

# Retrieve relevant context for a new analysis
context = rag.get_context_for_prompt(
    "authentication vulnerability in JWT token handling"
)
# Returns: chunks of text from previous reports most relevant to the query

RAGStore automatically enriches new analyses with context from previous runs, enabling Odin to:

  • Avoid duplicate analysis work on the same codebase
  • Cross-reference findings across different versions
  • Build institutional memory of analyzed repositories
