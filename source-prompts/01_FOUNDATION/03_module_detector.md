## 分析任务

### 1. 模块识别

根据以下线索识别模块：
- 顶级目录（每个目录可能是一个模块）
- 语言级别的命名空间/包（Python: package, JavaScript: module, Go: package, C: 文件组织）
- 功能划分（controller / service / model / view 等）
- 安全模块（auth / security / crypto / acl 等）

列出所有识别到的模块：

| 模块名 | 层级 | 职责 | 核心文件 | 安全相关 |
|--------|------|------|----------|---------|
| auth | src/auth/ | 用户认证 | login.c, jwt.c | ✓ |
| security | src/security/ | 安全机制 | acl.c, crypto.c | ✓ |
| ... | ... | ... | ... | |

### 2. 模块职责描述

对每个主要模块，给出：

- **模块名称**
- **核心职责**（一句话）
- **核心函数/文件**（2-3 个）
- **公开接口**（模块对外暴露的 API）
- **内部实现**（核心逻辑在哪里）
- **安全相关**（是否涉及安全敏感操作）

### 3. 依赖关系图

用 ASCII 字符绘制模块之间的依赖关系：

```
auth ──→ user ──→ database
  │                      ↑
api ──→ service ─────────┘
  │
router ──→ middleware

security (独立模块，无外部依赖)
  ↑
所有模块都可能依赖 security
```

### 4. 安全模块依赖分析

特别关注安全相关模块的依赖关系：

| 安全模块 | 依赖 | 被依赖 | 安全影响 |
|---------|------|--------|---------|
| crypto | 无 | auth, network | 加密模块被广泛信任 |
| acl | database | api | 鉴权模块被 api 依赖 |
| input_validator | 无 | 所有输入处理模块 | 广泛的基础模块 |

哪些模块依赖了安全模块？这些依赖是否合理？

### 5. 依赖方向分析

回答以下问题：
- 哪些模块是被其他模块依赖的（基础设施层）？
- 哪些模块依赖最多（业务核心层）？
- 是否存在循环依赖？
- 依赖方向是否清晰（单向依赖还是双向）？
- **安全模块是否被正确隔离**？

### 6. 聚合度与耦合度评估

| 模块 | 内聚度（高/中/低） | 耦合度（高/中/低） | 安全耦合 | 说明 |
|------|-------------------|-------------------|---------|------|
| auth | 高 | 中 | 高 | 涉及敏感操作 |
| crypto | 高 | 低 | 极高 | 不依赖其他模块 |
| ... | ... | ... | ... | ... |

**安全耦合**：模块间的安全依赖是否合理？是否有绕过安全检查的风险？

### 7. 架构观察

- 是否有明显的分层？
- 分层是否合理？
- 安全模块是否独立于业务逻辑？
- 是否有需要重构的地方？
