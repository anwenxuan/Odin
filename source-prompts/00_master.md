# 00 — 源码研究总控协议（Master Protocol）

> **版本**：v3.0（新增）
> **作用**：作为 source-prompts v2.3 的统一入口，定义阶段调度、证据标准、状态管理与收敛判定。01-32 为本总控的能力插件。
> **输入**：GitHub 仓库（README + 目录树 + 代码文件）
> **输出**：阶段性研究报告 + 多轮状态文件

---

## 核心目标

- 源码级调用链分析
- 问题驱动的深度追踪
- 漏洞假设与验证闭环
- 多轮可累积研究

---

## 术语定义

| 术语 | 定义 |
|------|------|
| **MEU** | 最小证据单元：文件路径、函数签名、分支条件、数据结构字段、调用关系，满足 3/5 项视为有证据 |
| **状态** | confirmed（证据充分且无反例）/ hypothesis（证据不完整）/ refuted（存在反例）/ unknown（当前无法回答） |
| **调用链最小元素** | 入口点 → 解析层 → 分发函数 → 核心逻辑 → 副作用点（含失败路径） |
| **收敛条件** | 所有高优先级问题 confirmed；或剩余 unknown ≤ 2 且需要特殊材料；或 research budget 用尽 |

---

## 总控状态声明（每轮必须读取并更新）

```yaml
round: N
project_type: [web服务/CLI/库/中间件/分布式/其他]
security_focus: [高危模块列表]
state:
  confirmed: [结论列表]
  hypothesis: [结论列表]
  refuted: [结论列表 + 反证]
  unknown: [问题列表 → 下一轮优先级]
open_questions: [从 hypothesis + unknown 提取，待本轮攻克]
depth_tracked: [问题ID → 已追踪层数，建议上限 5 层]
```

---

## 研究阶段（严格按顺序执行）

### 阶段 0：初始化（Round 1 入口）

**触发条件**：首次研究该项目，无状态文件

**动作**：
1. 执行 `01_repo_scanner`，输出项目类型标签 + 技术栈清单 + 安全敏感区域
2. 执行 `03_module_detector`，输出模块清单 + 依赖关系图
3. 执行 `04_entry_point_finder`，输出入口清单 + 初始化流程
4. 初始化总控状态文件（round: 1，state 全为空）
5. 进入阶段 1

---

### 阶段 1：问题生成（每轮必须执行）

**前置**：阶段 0 或上一轮阶段 8 未收敛

**动作**：
1. 读取当前状态文件中的 `open_questions`
2. 执行 `31_question_generator`（增强版），生成 3-7 个高优先级问题
3. 为每个问题标注安全映射标签（如 `#SECURITY:08`）
4. 将新问题合并到 `open_questions`
5. 按优先级排序，截取前 7 个（P0 优先）

**完成判定**：生成 3-7 个问题，每个问题有起点函数 + 终点状态 + MEU 起点条件

---

### 阶段 2：调用链追踪

**前置**：阶段 1 输出的问题清单

**动作**：
1. 对 `open_questions` 中的每个问题执行调用链追踪
2. 执行 `20_call_graph_dataflow_analyzer` + `18_api_config_analyzer`
3. 强制输出：
   - 完整调用链（主路径 + 至少一条异常/失败路径）
   - 关键函数卡片（函数名 + 文件 + 签名 + 调用者 + 被调用者 + 副作用）
   - 关键数据结构卡片（字段 + 指针关系 + 复杂度）
4. 更新 `depth_tracked`（已达 5 层的标记截断）
5. 对每个问题的结论执行 MEU 判定

**MEU 判定规则**：

```
IF 结论 MEU < 3/5:
    → 输出【证据不足】+ 需要补充的证据类型
    → 状态标记为 hypothesis

ELIF 存在反例:
    → 输出【反证】+ 反例代码路径
    → 状态标记为 refuted

ELIF MEU >= 3/5 且无反例:
    → 输出【结论】+ 完整 MEU 清单
    → 状态标记为 confirmed
```

**完成判定**：每个 `open_questions` 中的问题至少追踪到 3 层调用深度

---

### 阶段 3：安全分析与漏洞假设

**前置**：阶段 2 的调用链输出 + `state.hypothesis` 列表

**动作**：
1. 对每个 `hypothesis` 结论执行攻击面识别
2. 按安全映射标签调用对应安全分析模块（07-16）
3. 识别关键风险点：
   - 未验证输入路径
   - 内存操作边界
   - 认证/加密薄弱环节
   - 并发竞态窗口
   - 配置注入面
4. 生成漏洞假设（每个假设必须包含：触发条件 + 前置要求 + MEU）
5. 更新 `state.hypothesis` 列表

**完成判定**：每个 hypothesis 都有对应的安全分析输出

---

### 阶段 4：漏洞验证与 PoC

**前置**：阶段 3 的漏洞假设列表

**动作**：
1. 对每个 `hypothesis`（漏洞假设）执行 PoC 构造或反证分析
2. 执行 `15_exploit_path_tracker`
3. 尝试构造可利用路径（若无法利用，给出不可利用的具体原因）
4. 执行 MEU 判定，更新状态：
   - PoC 有效 + MEU >= 3/5 → confirmed
   - 存在反例（条件不满足/利用代价过高）→ refuted
   - MEU < 3/5 → hypothesis（保留，进入下一轮）
5. 对每个 refuted 记录反例代码路径

**完成判定**：每个漏洞假设都有 confirmed / refuted / hypothesis 状态

---

### 阶段 5：研究综合

**前置**：阶段 4 所有状态更新完成

**动作**：
1. 汇总所有模块输出，按 8 章节统一 schema 格式化
2. 更新 `state`：
   - confirmed：进入最终报告
   - hypothesis：合并到 `open_questions`
   - refuted：归档到排除列表
   - unknown：合并到 `open_questions`（标注需要什么材料）
3. 生成 Round N 研究报告
4. 检查收敛条件

---

### 阶段 6：多轮收敛判定

**前置**：阶段 5 综合完成

**判定顺序**：

```
IF 所有 P0 高优先级问题 confirmed:
    → 研究收敛，输出最终报告，结束

ELIF 剩余 unknown <= 2 且需要特殊材料（二进制/运行态）:
    → 研究收敛（自然边界），输出最终报告，结束

ELIF research budget 用尽（已执行 N 轮，N 由用户指定）:
    → 资源耗尽收敛，输出当前状态 + 最终报告，结束

ELSE:
    → round += 1
    → 从阶段 1 重新进入
```

## 统一输出 Schema（8 章节）

每个阶段和每个模块的输出必须嵌入以下结构：

```markdown
# [研究问题编号] 问题标题
> 状态: confirmed / hypothesis / refuted / unknown
> 安全映射: #SECURITY:XX（按需）

## 调用链

### 主路径
1. [文件:行] 函数签名 → ...
2. [文件:行] 函数签名 → ...

### 异常/失败路径
- [文件:行] 分支条件 → 失败路径描述

## 关键函数
- `文件:行` 函数签名 | 调用者 | 被调用者 | 副作用

## 关键数据结构
- `结构体名` (文件:行)
  字段:
    - field1: type // 用途
    - field2: *struct // 指针关系
  复杂度: O(?)

## 证据（MEU）
- 文件:行 | 函数/字段/分支/调用关系 | 描述

## 设计决策与权衡
- 决策点 | 选型 | 替代方案 | 权衡理由

## 漏洞假设与验证
假设: ...
触发条件: ...
前置要求: ...
MEU 清单: [文件:行, 函数, 分支条件, 数据结构字段, 调用关系]
验证结果: hypothesis / confirmed / refuted
理由: ...

## 未解问题（下一轮优先级）
- [类型] 问题描述 | 需要什么证据 | 优先级

## 多轮状态更新
confirmed: [结论列表]
hypothesis: [结论列表]
refuted: [结论列表 + 反证路径]
unknown: [问题列表 → 下一轮优先级]
```

## 质量门控规则

### 防泛化规则

1. **禁止无证据结论**：任何结论必须嵌入 MEU，不满足则强制 hypothesis
2. **禁止占位符**：调用链必须使用实际函数名，禁止 `xxx_function()`
3. **禁止模糊表述**：`通常`、`一般`、`可能` 等词汇出现时，必须同时给出 MEU
4. **禁止臆造**：数据结构字段必须与源码定义一致，禁止臆造字段
5. **深度截断**：超过 5 层调用深度，强制截断并标注"需手动验证"

### 质量检查清单（每轮输出前必须自检）

- [ ] 每个 `open_questions` 都有起点函数 + 终点状态
- [ ] 每个问题至少追踪 3 层调用深度
- [ ] 每个结论都有 MEU（至少 3/5 类）
- [ ] 每个 hypothesis 都有触发条件 + 前置要求
- [ ] 每个 refuted 都有反例代码路径
- [ ] `state` 四类状态都已更新
- [ ] 收敛条件已检查

## 注意事项

1. **总控优先**：任何模块的局部指令与总控冲突时，以总控为准
2. **材料决定质量**：提供的源码材料越完整，AI 分析越准确
3. **分步深入**：不要一次性提供太多材料，分步使用效果更好
4. **研究有边界**：通过 research budget 和收敛条件控制研究深度
5. **状态持久化**：每轮结束后必须更新状态文件，作为下一轮的输入
6. **MEU 不可绕过**：在任何情况下，禁止在无 MEU 的情况下输出 `confirmed`

**注意**：本总控协议定义了整个研究框架的运行规则。01-32 各模块作为能力插件，由本总控按阶段调度。各模块的输出必须符合 8 章节统一 schema，以支持多轮累积与自动拼接。
