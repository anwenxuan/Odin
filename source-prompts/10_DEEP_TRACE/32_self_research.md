# 32 — 自驱代码研究与逐题解答器（增强版）

> **版本**：v3.0（增强）
> **作用**：基于问题清单（来自 31_question_generator 增强版），AI 自驱逐题读源码并给出可验证答案。本版本新增多轮状态机协议、MEU 判定规则、8 章节统一 schema，与 `00_master.md` 总控协议深度对齐，作为研究综合层的主要载体。
> **输入**：问题清单（来自 31，增强版）+ 代码目录 + 关键文件 + 当前状态文件（Round N > 1）
> **输出**：逐题答案（8 章节格式）+ 调用链 + 分支矩阵 + 过渡状态 + 证据 + 状态更新 + 多轮研究报告

---

## 与总控协议的衔接

- **在研究流程中的位置**：阶段 5（研究综合），与 `16_security_report`（增强版）共同作为综合层输出
- **主要职责**：对 `open_questions` 中的每个问题执行调用链追踪 → MEU 判定 → 状态更新
- **输入衔接**：接收阶段 2 的追踪结果 + 阶段 3 的漏洞假设，合并到统一状态管理
- **输出衔接**：输出 8 章节格式 + 状态更新，传递给阶段 6（收敛判定）
- **状态管理**：每轮结束时更新总控状态文件（`confirmed/hypothesis/refuted/unknown`）

---

## 术语与状态定义

| 状态 | 定义 | 标记时机 |
|------|------|---------|
| `confirmed` | MEU ≥ 3/5 类且无反例 | 满足 MEU 且无法反驳 |
| `hypothesis` | MEU < 3/5，或 MEU ≥ 3/5 但未完成反证 | 证据不完整 |
| `refuted` | 发现反例证据，原假设不成立 | 存在反例路径 |
| `unknown` | 问题存在但当前阶段无法回答 | 缺少起点函数或关键材料 |

### MEU 判定规则（强制执行）

```
IF 结论 MEU < 3/5:
    → 输出【证据不足】+ 需要补充的证据类型
    → 强制标记为 hypothesis
    → 禁止输出 confirmed

ELIF MEU >= 3/5:
    → 尝试找反例
    → 有反例 → 输出【反证】+ 反例代码路径 → 标记为 refuted
    → 无反例 → 输出【结论】+ 完整 MEU 清单 → 标记为 confirmed
```

### 深度截断规则

- 每个问题追踪上限：**5 层调用深度**
- 超过 5 层 → 强制截断，输出"【截断】追踪深度已达上限，需手动验证或补充运行态信息"
- 被截断的问题 → 状态保持 `hypothesis`，进入下一轮

## 研究原则（必须遵守）

1. 结论必须由源码证据支撑，禁止猜测
2. 每题必须回答"从哪里开始，到哪里结束"
3. 每题至少给出一条可验证调用链（建议 >= 3 层）
4. 找不到证据要明确写"未定位到"，并给出下一步定位函数
5. 先覆盖核心路径，再补充分支和异常路径
6. **强制执行 MEU 判定**：不满足 3/5 不得输出 confirmed
7. **禁止无证据结论**：不能用"通常"、"一般"、"可能"等模糊表述代替证据
8. **禁止占位符函数**：调用链必须使用实际函数名，不允许 `xxx_function()`
9. **输出必须符合 8 章节 schema**：每个问题的答案必须嵌入统一格式

## 自驱研究流程（增强版）

### 阶段 A：输入校验与优先级排序

对问题清单执行四步：

1. 校验问题是否满足"必须读源码才能答"
2. 校验每个问题是否有起点函数（MEU 起点条件中的必须项）
3. 按优先级排序：
   - P0：请求生命周期、并发一致性、启动流程
   - P1：数据结构、调度机制、生命周期管理
   - P2：监控统计、边角机制
4. 为每题绑定"起点函数"和"终点状态"（已在 31 中定义，此处验证）

### 阶段 B：逐题追踪与解答

每题按以下步骤执行：

1. 从起点函数进入
2. 跟踪主路径调用链（至少 3 层）
3. 识别关键分支条件（正常/异常/边界）
4. 提取过渡状态（并发切换、重建、重试、写缓冲）
5. **执行 MEU 判定**（5 类证据元素，≥3/5 才可 confirmed）
6. **尝试找反例**（针对假设性结论）
7. 给出结论并附源码证据 + MEU 清单
8. 更新状态：`confirmed` / `hypothesis` / `refuted`

### 阶段 C：交叉校验

1. 用指标更新点验证路径结论（例如计数器在哪更新）
2. 用生命周期函数验证资源结论（创建/持有/销毁）
3. 用启动流程验证时序结论（初始化顺序是否自洽）

### 阶段 D：状态汇总与更新

1. 汇总所有问题的状态
2. 输出 8 章节格式的研究报告
3. 更新总控状态文件（追加本轮结论）
4. 生成 Round N 研究报告
5. 输出下一轮 `open_questions`

## 与 00_master.md 的接口约定

- **输入**：
  - `open_questions`：来自 31_question_generator 的问题清单
  - 上一轮 `state.hypothesis`：未闭环的漏洞假设

- **收敛建议**：在最终报告中标注是否满足收敛条件，供阶段 6 判断

## 深度追踪规范（防止碎片化）

调用链必须包含以下 5 类元素，缺少任一类须在输出中明确标注：

| 元素类别 | 说明 | 示例 |
|---------|------|------|
| **入口点** | 外部数据进入系统的第一个函数 | `readQueryFromClient` |
| **解析层** | 将原始数据转换为内部表示的函数 | `processInputBuffer` |
| **分发函数** | 将请求路由到具体处理逻辑的函数 | `processCommand` |
| **核心逻辑** | 执行主要业务/安全检查的函数 | `expireIfNeeded` |
| **副作用点** | 产生可观察副作用的函数（写内存/发网络/写磁盘） | `afterCommand` |

> 若某类元素不存在于该问题的路径中，标注"无"并说明原因。

**注意**：
- 输出语言必须为中文
- 不要只给抽象解释，必须包含函数级路径和 MEU 证据
- 如果项目不是事件驱动，可将调度问题替换为任务执行框架/作业编排机制
- **MEU 判定不可绕过**：在任何情况下，禁止在无 3/5 类 MEU 的情况下输出 confirmed
- 状态更新必须完整：confirmed / hypothesis / refuted / unknown 四类必须全部包含
