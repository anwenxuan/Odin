# 29 — 事件/信号/扩展点系统分析

**作用**：分析框架的观察者模式和可扩展性机制，理解插件如何注入逻辑

**输入**：信号/事件定义文件 + 事件触发点代码 + 扩展点注册代码

**输出**：扩展点清单 + 触发时机表 + 插件注入路径

---

## 核心分析问题

1. 框架提供了哪些扩展点？它们的触发时机是什么？
2. 框架使用的是"拦截器链"（顺序执行，可短路）还是"事件广播"（并行执行，不可短路）？
3. 扩展点的注册和查找优先级是怎样的？
4. 插件如何利用扩展点注入逻辑？

---

## 分析任务

### 1. 扩展点类型识别

框架的扩展机制通常有三种形式，需全部识别：

| 扩展类型 | 执行模型 | 短路能力 | 典型框架/实现 |
|---------|---------|---------|-------------|
| **拦截器链（Hook）** | 顺序遍历，可返回值中断 | **可** | before_request、Django MIDDLEWARE |
| **事件广播（Signal/Event）** | 并行或顺序通知所有监听器 | **不可** | blinker、Spring ApplicationEvent、EventBus |
| **中间件链（Middleware）** | 洋葱模型，层层包装 | **可（通过短路响应）** | Express.js、ASP.NET Core |

**识别特征**：

```python
# 拦截器链：返回非 None 值短路
def before_request():
    if not auth: return 401_response  # 短路
    return None  # 继续
```

```python
# 事件广播：发送后不管结果
event_bus.publish("request.started", data)
# 所有监听器被通知，不影响主流程
```

```python
# 中间件链：层层包装
def middleware(next_handler):
    def wrapper(request):
        # 前置处理
        response = next_handler(request)
        # 后置处理
        return response
    return wrapper
```

### 3. 拦截器链分析

**注册表结构**：

```python
# 典型实现：按注册顺序存储的函数列表
before_request_funcs: dict[ScopeKey, list[Handler]]
# ScopeKey 可以是：全局、应用、蓝图、路由等层级
```

**执行模型**：

```
for scope in [全局, 蓝图, 路由]:
    for func in before_request_funcs[scope]:
        rv = func()
        if rv is not None:
            return rv  # 短路：非 None 返回值作为响应
```

**优先级链**：当同一扩展点有多个注册者时，分析优先级规则

| 优先级维度 | 规则 | 实现方式 |
|-----------|------|---------|
| 层级顺序 | 全局 → 蓝图 → 路由 | 按 key 顺序遍历 dict |
| 注册顺序 | 先注册先执行 | 列表顺序 |
| 显式优先级 | 可指定执行顺序 | `@hook(priority=100)` |

### 4. 事件广播分析

**发布-订阅模型**：

```python
# 发布者
event_bus.publish("event.name", sender=self, **kwargs)

# 订阅者
event_bus.subscribe("event.name", handler)
# 或装饰器
@event_bus.on("event.name")
def handler(sender, **kwargs): ...
```

**关键设计决策**：

| 问题 | 影响 | 框架选择 |
|------|------|---------|
| 异常是否向上传播 | 监听器崩溃是否中断主流程 | 吞掉 / 向上抛出 |
| 有无执行顺序保证 | 监听器之间的依赖关系 | 无保证 / 注册顺序 |
| 同步还是异步 | 监听器是否阻塞主线程 | 同步 / 队列异步 |
| 有无优先级 | 监听器执行先后 | 无 / priority 参数 |

### 5. 插件注入路径

分析插件利用扩展点的常见方式：

| 注入方式 | 实现原理 | 适用场景 | 典型框架 |
|---------|---------|---------|---------|
| 信号监听 | `signal.connect()` | 日志、监控、统计 | Flask (blinker) |
| 钩子注册 | `hooks.append(handler)` | 认证、预处理 | Django (MIDDLEWARE) |
| 扩展注册 | `app.extensions[name] = state` | 插件状态共享 | Flask-SQLAlchemy |
| 子类覆盖 | 继承 + 重写方法 | 功能增强 | Flask-Login |
| 中间件包装 | WSGI/框架中间件 | CORS、压缩、认证 | Express.js、Flask-CORS |
| 函数覆盖 | monkey patch | 修改默认行为 | 慎用 |

### 6. 扩展点的执行时序图

```
请求进入
    ↓
[中间件 1] ──前置处理──→ [中间件 2] ──前置处理──→ [拦截器链]
                                                              ↓
                                                        [业务处理]
                                                              ↓
[中间件 2] ←──后置处理── [中间件 1] ←──后置处理── [拦截器 after_xxx]
    ↓
事件广播 → [监听器1] → [监听器2] → ... （并行或顺序）
    ↓
响应发出
```

### 7. 扩展点的安全考量

- **拦截链中的短路**：认证失败直接返回 401，跳过后续处理
- **事件监听器的异常**：吞掉异常可能导致静默失败
- **插件执行顺序**：依赖注入的插件若顺序错误可能导致崩溃
- **中间件的顺序敏感**：CORS 中间件必须在认证中间件之前

**注意**：
- 本提示词适用于任何语言和框架（Python、Java、Go、Node.js 等）
- "事件/信号"是 Python 的常见术语，对应 Java 的 `ApplicationEvent`、JavaScript 的 `EventEmitter`、Go 的 channel 或 EventBus 库
- 拦截器和事件的根本区别：拦截器可以短路流程，事件不能
