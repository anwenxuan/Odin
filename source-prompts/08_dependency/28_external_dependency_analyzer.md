# 28 — 外部依赖边界分析

**作用**：识别框架/库中哪些能力是委托给第三方库的，评估可替换性和信任边界

**输入**：依赖声明文件 + 第三方库源代码 + 框架核心抽象代码

**输出**：依赖边界图 + 可替换性评估 + 信任边界说明

---

## 核心分析问题

1. 哪些核心能力是第三方库提供的，而不是自研的？
2. 框架与第三方库之间是否有抽象层（松耦合 vs 强耦合）？
3. 第三方库的漏洞或变更会多大程度影响框架本身？
4. 框架提供了哪些可替换点，允许用户换掉默认实现？

---

## 分析任务

### 1. 收集依赖信息

**依赖声明文件**：

| 语言 | 依赖文件 | 关键信息 |
|------|---------|---------|
| Python | `requirements.txt` / `pyproject.toml` / `setup.py` | 包名、版本约束 |
| Java | `pom.xml` / `build.gradle` | groupId、artifactId、scope |
| Go | `go.mod` / `go.sum` | module path、replace 指令 |
| C/C++ | `CMakeLists.txt` / `conanfile.py` / `vcpkg.json` | 库名、版本、引入方式 |
| Rust | `Cargo.toml` | crate name、version、features |
| Node.js | `package.json` / `yarn.lock` | package name、version range |

**分析步骤**：
- 列出所有直接依赖和间接依赖
- 标记出框架自身提供的模块（源码在项目中）
- 标记出外部依赖（源码在项目外）

### 2. 委托边界划分

对每个核心功能，判断其实现来源：

| 功能 | 来源 | 委托方式 | 可替换性 | 风险等级 |
|------|------|---------|---------|---------|
| [功能名] | [第三方库 / 自研] | [接口抽象 / 函数调用 / 硬编码] | [高/中/低] | [高/中/低] |

**委托类型识别**：

| 类型 | 特征 | 典型场景 |
|------|------|---------|
| **完全委托** | 核心能力完全交给第三方库 | Flask 路由 → Werkzeug；Gin 路由 → radix tree |
| **部分委托** | 核心抽象自研，底层委托 | JSON 序列化 → 可替换后端 |
| **适配器封装** | 第三方作为底层，上层做框架适配 | Django ORM → SQLAlchemy；Go stdlib net/http 包装 |
| **胶水代码** | 轻量集成，仅做接口转换 | 语言 binding、FFI wrapper |

### 3. 依赖层次分析

绘制项目的依赖拓扑：

```
┌─────────────────────────────────────────────────┐
│           框架核心层（自研代码）                   │
│   业务逻辑 · 扩展点（钩子/信号/中间件）· 抽象层接口  │
├─────────────────────────────────────────────────┤
│           第三方库层（外部依赖）                   │
│   路由解析 · HTTP 抽象 · CLI 框架 · 序列化库       │
├─────────────────────────────────────────────────┤
│           语言运行时层                            │
│   Python/Go/JVM · 标准库                         │
└─────────────────────────────────────────────────┘
```

### 4. 接口抽象评估

找出框架为降低耦合而设计的抽象层：

| 抽象接口 | 作用 | 可替换实现 | 替换代价 |
|---------|------|---------|---------|
| [接口名] | [描述] | [可替换的库/实现] | [高/中/低] |

**评估维度**：
- 是否有 `interface` / `Protocol` / `ABC` / `trait` 抽象
- 是否可以通过配置或参数替换
- 替换后是否需要修改框架核心代码
- 是否有多个内置实现供选择

### 5. 信任边界分析

明确框架代码与第三方库代码的分界线：

**每个第三方依赖需回答**：
- 这个库提供什么能力？（功能边界）
- 这个库可能出现什么风险？（漏洞、更新、弃用）
- 如果它出问题，框架能做什么？（fallback、隔离）

| 第三方库 | 信任级别 | 已知风险 | 框架的隔离措施 |
|---------|---------|---------|-------------|
| [库名] | [可信/条件可信/谨慎] | [漏洞类型/版本问题] | [抽象层/版本约束/监控] |

### 6. 可替换性评估

**替换性高的特征**：
- 有明确的接口抽象层
- 可以通过配置或构造函数注入
- 框架提供了多个内置选项

**替换性低的特征**：
- 硬编码直接调用
- 没有抽象层，直接依赖具体类
- 替换需要修改核心代码

---

## 输出：依赖边界图与可替换矩阵

### 依赖边界图

```
[自研核心] ──调用──> [抽象层接口] ──实现──> [第三方库 A]
                                        ├──> [第三方库 B]
[自研核心] ──直接调用──> [第三方库 C] （低可替换性）
```

### 可替换性矩阵

| 组件 | 当前依赖 | 可替换 | 替换方式 | 替换成本 |
|------|---------|-------|---------|---------|
| [组件] | [库] | [是/否] | [配置/继承/注入] | [低/中/高] |

---

## 实施步骤

1. **收集依赖清单**：读取项目的依赖声明文件，列出所有第三方依赖
2. **追踪核心导入**：从入口文件开始，追踪 `import` / `require` / `use` 语句，确定每个模块的来源
3. **划分委托边界**：对每个核心功能，判断是完全委托还是部分委托
4. **定位抽象层**：找出框架提供的接口抽象和可替换点
5. **评估信任边界**：分析每个第三方库的可信度和风险
6. **绘制依赖图**：用 ASCII 图展示依赖拓扑和信任边界

---

**注意**：
- 本提示词适用于任何语言和框架，不绑定特定技术栈
- 第三方库不等于不可信，但必须明确其能力和限制
- 可替换性高的依赖是框架灵活性的体现，也是安全隔离点
- 本提示词聚焦"分析"，安全相关的依赖漏洞检查参见 `08_input_validation_analyzer` 和 `14_config_security_analyzer`
